## 二、第一性原理分析

### 2.1 当前分类器的本质是什么？

```
输入: OCR文本列表 → 拼接为全文
处理: 对每种文档类型，检查是否存在关键词
输出: 匹配到多个类型时，按预定义优先级选最高的
```

**核心机制**: 关键词存在性检测 + 静态优先级裁决

### 2.2 这个方法的基本假设是什么？

1. **假设1**: 关键词能唯一标识文档类型
2. **假设2**: 文档中出现某个关键词，就意味着该文档属于那个类型
3. **假设3**: 静态优先级能正确裁决冲突

### 2.3 这些假设成立吗？

**假设1: ❌ 不成立**
- "身份证"出现在：身份证✅、结婚证✅、户口本✅、购房合同✅、房产证✅
- "登记日期"出现在：结婚证✅、户口本✅、购房合同✅、房产证✅
- 关键词不是唯一的，它们跨类型出现

**假设2: ❌ 不成立**
- 结婚证的备注区域包含"身份证件号"字段
- 购房合同的法律条款引用"居民身份证"
- 户口本有"登记事项变更"页面包含"登记日期"列
- 文档**提及**某事物 ≠ 文档**是**那个事物

**假设3: ❌ 不成立**
- 优先级1(身份证) > 优先级2(结婚证)，导致结婚证中的"身份证件号"触发身份证分类
- 优先级2(结婚证) > 优先级3(户口本)，导致户口本中的"登记日期"触发结婚证分类
- 静态优先级无法处理"强信号"和"弱信号"的区别

## 三、苏格拉底提问法：追问根因

### Q1: 为什么户口本被误判为结婚证？(9/12 = 75%错误率)

**A1**: 因为户口本和结婚证同时匹配了关键词，结婚证的优先级(2)高于户口本(3)。

**Q2: 户口本匹配了哪些关键词？结婚证匹配了哪些？**

**A2**:
- 户口本匹配了："户主" (因为页面有"户主姓名"字段)
- 结婚证匹配了："登记日期" (因为页面有"登记事项变更和更正记载"表头)
- 有些还匹配了："公民身份号码" (因为个人页有身份证号)

**Q3: 为什么"登记日期"会出现在户口本中？**

**A3**: 因为户口本的"登记事项变更和更正记载"页面是一个表格，表头有"登记日期"列名。这是行政表格的通用格式，不是结婚登记特有的。

**Q4: 户口本最独特的标识是什么？**

**A4**: **"常住人口登记卡"** — 这是户口本个人页的标题，其他任何文档都不会出现这个词。但当前规则中**完全没有这个关键词**！

**Q5: 为什么没有"常住人口登记卡"这个关键词？**

**A5**: 规则设计者使用了泛化的词("户口本"、"户口簿")，但实际文档中的措辞是更正式的"常住人口登记卡"。这是关键词设计与实际文档用语不匹配的问题。

**Q6: 所以根因是什么？**

**A6**:
1. **缺失强特征关键词**："常住人口登记卡"是户口本最强信号，但未被纳入
2. **弱关键词冲突**："登记日期"在多种文档中出现，不是好的区分特征
3. **优先级方向错误**：户口本的优先级(3)低于结婚证(2)，但"户主"比"登记日期"更具区分力

### Q7: 为什么结婚证被误判为身份证？(3/8 = 37.5%错误率)

**A7**: 因为结婚证备注区域包含"身份证件号"字段，触发了"身份证"关键词。

**Q8: 为什么"身份证"关键词会匹配到"身份证件号"？**

**A8**: 因为代码使用 `keyword in full_text` 子串匹配。"身份证"是"身份证件号"的子串。

**Q9: "身份证件号"和"公民身份号码"有什么区别？**

**A9**:
- "公民身份号码" → 出现在**身份证本体**上，是身份证的强特征
- "身份证件号" → 出现在**其他证件**的备注中，引用当事人的身份证号
- 前者说明"这是身份证"，后者说明"这不是身份证，但提到了身份证"

**Q10: 所以根因是什么？**

**A10**:
1. **关键词粒度过粗**："身份证"太短，会误匹配"身份证件号"、"身份证复印件"等
2. **缺少上下文区分**：无法区分"这是身份证本体"和"其他文档引用了身份证号"
3. **优先级过高**：ID_CARD优先级为1，一旦匹配就碾压其他类型

### Q11: 为什么购房合同被误判为身份证？(2/8 = 25%错误率)

**A11**: 合同条款中提到"买受人应提供居民身份证复印件"等文字。

**Q12: 合同中明明有"商品房买卖合同"这个强特征词，为什么没生效？**

**A12**: 因为"商品房买卖合同"匹配了PURCHASE_CONTRACT(优先级6)，但"身份证"匹配了ID_CARD(优先级1)，优先级1碾压了优先级6。

**Q13: 这说明了什么？**

**A13**: **当前优先级体系完全颠倒了！** 应该是：
- "商品房买卖合同" → 强信号，几乎100%确定是购房合同
- "身份证"出现在合同文本中 → 弱信号，不能说明这是身份证

但系统给了弱信号更高的优先级。

### Q14: 有些图片标签正确吗？

**A14**: 部分样本标签可能不准确：
- `027aa4151f75424c8529755ada5e6064.jpg` 标签为id_card_front，但云端看到的是户口本"登记事项变更"页
- `e9505e57b2d646b9a06c121fe96a8a4a.jpg` 标签为property，但云端看到的是"存量房买卖合同"
- 一个case文件夹包含多种文档，样本选取时可能标错了页面类型

## 四、根因总结

```
问题表现: 分类器准确率仅30%
│
├── 根因1: 关键词不是唯一标识符 (关键词重叠)
│   ├── "身份证" 出现在 6种文档中
│   ├── "登记日期" 出现在 4种文档中
│   └── "户主" 同时出现在户口本和结婚证的上下文中
│
├── 根因2: 子串匹配产生假阳性
│   ├── "身份证" 匹配 "身份证件号"(其他文档引用)
│   ├── "登记日期" 匹配 "抵押登记日期"(合同条款)
│   └── "存量房" 匹配 "存量房交易资金监管"(监管协议)
│
├── 根因3: 优先级体系有根本缺陷
│   ├── 强信号("商品房买卖合同")优先级6(最低)
│   ├── 弱信号("身份证")优先级1(最高)
│   └── 优先级是静态的，无法根据上下文调整
│
├── 根因4: 关键强特征词缺失
│   ├── 户口本缺少"常住人口登记卡"
│   ├── 房产证缺少"不动产权证书 BDCQZ"
│   └── 合同缺少"GF-2014-0171"等标准编号
│
└── 根因5: 部分测试样本标签可能有误
    └── case文件夹含多种文档，页面类型标错
```

## 五、解决方案对比

### 方案A: 修复关键词 + 调整优先级（最小改动）

**做法**:
- 添加"常住人口登记卡"为户口本最高优先关键词
- 将"身份证"改为"公民身份号码"或要求精确匹配
- 调整优先级：HOUSEHOLD_REGISTER > MARRIAGE_CERTIFICATE

**优点**: 改动最小，立即可做
**缺点**: 治标不治本，未来新文档类型还会遇到类似问题
**预期效果**: 准确率从30%→50-60%

### 方案B: 强/弱关键词分级

**做法**:
- 将关键词分为"强信号"(唯一标识)和"弱信号"(辅助判断)
- 强信号匹配 → 直接确定类型，不参与优先级竞争
- 弱信号匹配 → 参与评分，但不覆盖强信号
- 示例：
  - 强信号: "常住人口登记卡"→户口本, "商品房买卖合同"→购房合同, "公民身份号码"→身份证, "结婚证字号"→结婚证
  - 弱信号: "身份证"(去掉或降级), "登记日期"(去掉或降级)

**优点**: 解决了强弱信号混淆的核心问题
**缺点**: 需要仔细定义每个关键词的强弱等级
**预期效果**: 准确率从30%→70-80%

### 方案C: 多信号加权评分

**做法**:
- 每个关键词有权重(基于信息熵/区分力)
- 对每种文档类型计算加权得分
- 选择得分最高的类型
- 需要设定最小阈值，低于阈值判为UNKNOWN

**优点**: 最灵活，能处理复杂的关键词重叠
**缺点**: 权重需要训练/调参，复杂度高
**预期效果**: 准确率从30%→80-90%

### 方案D: 两阶段分类（推荐）

**做法**:
- **阶段1: 强信号快速判定**
  - 检查是否存在"决定性关键词"（如"常住人口登记卡"、"商品房买卖合同"）
  - 如果匹配 → 直接返回，不参与后续竞争
- **阶段2: 多信号评分**
  - 如果没有强信号 → 使用加权关键词评分
  - 关键词权重基于其区分力（出现在越少文档类型中，权重越高）
- **阶段3: 不确定时降级到VLM**
  - 如果评分差距很小 → 交给VLM模型判断

**优点**: 
- 结合了速度和准确性
- 强信号秒判，弱信号精判
- 有降级机制兜底
**缺点**: 实现复杂度中等
**预期效果**: 准确率从30%→85-95%

### 方案E: 使用VLM分类

**做法**:
- 完全跳过关键词分类
- 用VLM模型（GLM-OCR或云端模型）直接判断文档类型
- 给出分类prompt让模型返回类型

**优点**: 最准确，能处理任何文档类型
**缺点**: 慢（每张图片需要10-30秒VLM推理），成本高
**预期效果**: 准确率95%+

## 六、推荐方案

**推荐: 方案D（两阶段分类）为主，融合方案B（强弱关键词分级）**

理由:
1. 解决了根本问题（强弱信号混淆），而不是仅仅"调参数"
2. 强信号快速判定保证了速度
3. 评分机制处理了边界情况
4. 降级到VLM提供了兜底
5. 架构可扩展，新增文档类型只需添加强/弱信号定义

### 具体实施步骤

1. **定义强信号关键词表** (决定性标识)
   ```python
   STRONG_SIGNALS = {
       DocumentType.HOUSEHOLD_REGISTER: ["常住人口登记卡", "居民户口簿首页"],
       DocumentType.ID_CARD: ["公民身份号码"],  # 仅限身份证号本身
       DocumentType.MARRIAGE_CERTIFICATE: ["结婚证字号"],
       DocumentType.PURCHASE_CONTRACT: ["商品房买卖合同"],
       DocumentType.PROPERTY_CERTIFICATE: ["不动产权证书"],
   }
   ```

2. **定义弱信号关键词表** (辅助判断)
   ```python
   WEAK_SIGNALS = {
       DocumentType.ID_CARD: ["姓名", "性别", "民族", "出生", "住址"],
       DocumentType.MARRIAGE_CERTIFICATE: ["持证人", "男方", "女方"],
       DocumentType.HOUSEHOLD_REGISTER: ["户主姓名", "户号", "户别"],
       ...
   }
   ```

3. **实现两阶段分类逻辑**
   - 阶段1: 检查强信号 → 匹配到直接返回
   - 阶段2: 弱信号加权评分 → 选最高分
   - 阈值: 分数差距 < 0.5时返回UNCERTAIN，降级到VLM

4. **移除有害关键词**
   - 删除 "身份证" (太泛) → 只保留 "公民身份号码"
   - 删除 "登记日期" (太泛) → 改为 "结婚证字号" + "登记日期" 组合判定
   - 删除 "权利人" → 改为 "不动产权证书" + "权利人" 组合判定
