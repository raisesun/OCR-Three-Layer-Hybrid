## 1. 问题本质（一句话）

`_remove_ocr_char_spacing` 判定"字间空格模式"后用 `re.sub(r"\s+", "", line)` 移除**所有**空格。当 OCR 在同一行输出多个字段且都呈字间空格模式时，**字段间分隔空格也被移除，多个字段被错误合并成一个字符串**。

---

## 2. 事实链（代码层面，已逐行验证）

### 2.1 调用位置（text_preprocessor.py:77-85）
`preprocess_text` 步骤 4 对每行调 `_remove_ocr_char_spacing`：
```python
lines = result.split("\n")
for line in lines:
    processed_lines.append(self._remove_ocr_char_spacing(line))
```

### 2.2 _remove_ocr_char_spacing 逻辑（:100-141）
```python
parts = line.split()                          # 按空格分割
# 统计单字符部分（content_chars <= 1）vs 多字符部分
for part in parts:
    content_chars = re.findall(r"[一-鿿\d]", part)
    if len(content_chars) <= 1: single_char_parts += 1
    else: multi_char_parts += 1

ratio = single_char_parts / total
if ratio >= 0.8 and single_char_parts >= 3:
    return re.sub(r"\s+", "", line)           # 移除所有空格
else:
    return line.strip()
```

### 2.3 H6 触发场景
OCR 输出一行：`"合 同 编 号 ： 2 0 2 4 买 方 ： 张 三"`
- parts = `["合","同","编","号","：","2","0","2","4","买","方","：","张","三"]`
- 每个 part 单字符（"合"/"2"）或单标点（"："，content_chars=[]，len=0 <=1 算单字符）
- single=14, multi=0, ratio=1.0 >= 0.8, single>=3 -> 触发
- `re.sub(r"\s+", "", line)` -> `"合同编号：2024买方：张三"`
- **"合同编号：2024" 和 "买方：张三" 两个字段被合并**

### 2.4 正常场景（不受影响）
- `"商 品 房 买 卖 合 同"` -> parts 全单字符 -> 移除空格 -> `"商品房买卖合同"` ✅ 正确
- `"姓名 张三"` -> parts ["姓名"(多字符),"张三"(多字符)] -> ratio=0 < 0.8 -> 保留 ✅ 正确

## 3. 根因分析（深入）

### 3.1 为什么会合并？
算法判定"字间空格模式"的唯一标准是"≥80% 部分是单字符"。但这个标准**无法区分**：
- **字间空格**：`"合 同 编 号"`（一个字段内部，字符间空格）-- 应移除
- **多字段同行字间空格**：`"合 同 编 号 ： 2 0 2 4 买 方 ： 张 三"`（多个字段，每字段内部字间空格）-- 字段内空格应移除，字段间空格应保留

两者都满足"全单字符"，算法一律移除所有空格，导致多字段合并。

### 3.2 苏格拉底追问
- **Q：为什么不区分字段边界？**
  -> 算法只看字符级统计，不识别"标签：值"结构。没有字段分隔符的概念。
- **Q：冒号不是字段分隔符吗？**
  -> 冒号"："在 parts 里是单标点部分，被当作普通单字符，不特殊处理。
- **Q：OCR 真的会输出这种格式吗？**
  -> 取决于 OCR。PP-OCRv6 通常按文本块输出（一行一字段），但紧凑布局（表格行含多字段）可能一行多字段。且字间空格模式本身是某些 OCR（尤其低质量扫描）的产物。**触发概率需运行时确认**。
- **Q：是设计缺陷还是边界遗漏？**
  -> 边界遗漏。字间空格移除对单字段正确，但没考虑多字段同行的边界情况。

## 6. 修复方案

### 方案 A：冒号分段处理（推荐）
按全角冒号"："分段，每段单独移除字间空格，再连接：
```python
if "：" in line or ":" in line:
    # 按冒号分段，每段单独处理
    segments = re.split(r"([：:])", line)  # 保留冒号
    processed = [self._remove_ocr_char_spacing(seg) if i % 2 == 0 else seg
                 for i, seg in enumerate(segments)]
    return "".join(processed)
# 无冒号，原逻辑
```
- 优点：保留字段边界（冒号分隔）
- 缺点：`"2 0 2 4 买 方"`（冒号间的值+下个标签）仍全单字符，移除后 "2024买方" 仍合并

### 方案 B：保守--多冒号行不处理
行内冒号 >= 2 时，不执行全空格移除（只 strip）：
```python
if line.count("：") + line.count(":") >= 2:
    return line.strip()  # 多字段行，保留空格避免合并
# 原逻辑
```
- 优点：简单、避免合并、零回归风险（正常单字段场景不变）
- 缺点：多字段行的字间空格保留（可能影响正则，但比合并好）

### 方案 C：只移除单字符部分之间空格
遍历 parts，合并相邻单字符部分，遇多字符部分或冒号时分隔：
- 优点：精准
- 缺点：实现复杂，边界多

### 方案 D：移除空格后在冒号后恢复分隔
`re.sub` 后在"："后插入空格：
- 缺点："合同编号：2024"的冒号后不该加空格（值紧跟），逻辑复杂

---

## 7. 推荐方案

**推荐方案 B（保守：多冒号行不处理）**，理由：
1. **零回归风险**：正常单字段字间空格场景（0-1 冒号）不变
2. **避免合并**：多字段行（>=2 冒号）保留空格，不合并
3. **简单**：1 行判断
4. **trade-off 可接受**：多字段行保留字间空格，正则可能受影响，但比合并（正则必失败）好；且有 VLM 兜底保底

**实施要点**（text_preprocessor.py:137 前）：
```python
# 多字段行（>=2 冒号）不执行全空格移除，避免合并字段（H6）
if line.count("：") + line.count(":") >= 2:
    return line.strip()

ratio = single_char_parts / total
if ratio >= 0.8 and single_char_parts >= 3:
    return re.sub(r"\s+", "", line)
else:
    return line.strip()
```

**测试**：
- 加多字段场景：`"合 同 编 号 ： 2 0 2 4 买 方 ： 张 三"` -> 应保留空格（不合并）
- 回归正常字间空格：`"商 品 房"` -> "商品房"（0 冒号，正常移除）
- 回归 `"姓名 张三"` -> 保留（1 冒号无，ratio<0.8）
