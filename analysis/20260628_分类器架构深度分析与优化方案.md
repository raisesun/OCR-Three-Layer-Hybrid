## 二、第一性原理分析

### 2.1 问题的本质

文档类型识别 = **模式匹配问题**，有两条路径：
- **规则匹配**：快（微秒级）、但不完整
- **视觉理解**：慢（秒级）、但全面

### 2.2 系统架构

当前三层架构：
```
分类器（关键词匹配）→ 路由层（按类型分发）→ 提取层（RULE/VLM/LLM）
```

当分类器返回UNKNOWN时，文档进入VLM层，由VLM同时做分类+提取。

## 三、苏格拉底提问法：连续追问

### 第一层：分类器的目的是什么？

**Q1: 为什么需要分类器？**
> A1: 将文档路由到正确的提取器，不同类型的文档需要提取不同的字段。

**Q2: 分类器必须100%准确吗？**
> A2: 理想状态下应该准确，但如果分类器返回UNKNOWN，VLM层可以处理。

**Q3: VLM层能正确处理UNKNOWN吗？**
> A3: 能，但这增加了VLM层的职责复杂度（同时做分类+提取）。

**结论1：** 分类器的目的是**快速分流**，让大多数文档走规则提取的快速通道。

---

### 第二层：我们的目标是什么？

**Q4: 我们的最终目标是什么？**
> A4: 构建一个准确、高效、可维护的不动产文档OCR系统。

**Q5: 分类准确率和提取准确率的关系是什么？**
> A5: 分类是提取的前提。分类错误 → 路由到错误的提取器 → 提取失败。

**结论2：** 分类准确率直接影响提取准确率。分类器应该尽可能多地确定文档类型。

---

### 第三层：数据分布告诉我们什么？

**Q6: 25个错误案例的根因是什么？**
> A6:
> - 18个可通过优化规则解决
> - 7个是真正的边界情况（附属页面、OCR质量差等）

**Q7: 这18个"可优化"案例说明什么？**
> A7: 分类器的规则覆盖不够全面，只覆盖了"标准情况"，没有覆盖"变体情况"。

**结论3：** 分类器规则需要扩展，覆盖更多变体。

---

### 第四层：VLM应该在哪里介入？

**Q8: VLM的优势和劣势是什么？**
> A8:
> - 优势：视觉理解能力，能处理模糊/低质量图片、复杂版面
> - 劣势：推理速度慢（秒级）

**Q9: 在我们的场景中，速度重要吗？**
> A9: VLM是本地部署（qwen3.5-4B），经济成本≈0。时间成本约1-2秒/张，评估场景可接受。

**Q10: 既然成本不是问题，为什么不充分利用VLM？**
> A10: 应该让VLM处理更多场景。

**Q11: VLM应该在哪个阶段介入？**

| 方案 | 描述 | 准确率 | 职责清晰度 | 延迟 |
|------|------|--------|-----------|------|
| A. 当前设计 | UNKNOWN → VLM同时分类+提取 | 85% | 低 | 低 |
| B. VLM分类兜底 | 规则失败→VLM分类→路由提取 | 95%+ | 高 | 中 |

**结论4：** 应该让VLM在分类阶段就介入，获得更高的准确率和更清晰的架构。

## 四、批判性思维：挑战假设

### 挑战1：UNKNOWN真的是问题吗？

**反思：** 如果VLM能处理UNKNOWN，为什么还要优化分类器？

**分析：**
- UNKNOWN → VLM → 分类+提取 → 可能准确
- 但如果VLM的分类错了呢？提取也会错
- 分类器的作用是**减少VLM的分类负担**，提高整体准确率

**结论：** UNKNOWN本身不是问题，问题是VLM的分类准确率是否足够高。

---

### 挑战2：附属页面应该怎么处理？

**反思：** 宗地图/附图应该被分类为"不动产权证书"吗？

**分析：**
- 如果分类为"不动产权证书" → 提取字段会失败（没有权利人等信息）
- 如果分类为UNKNOWN → VLM判断为附属页面 → 跳过提取

**结论：** 附属页面应该在分类阶段被识别并标记，不参与字段提取。

## 五、推荐方案：VLM分类兜底

### 5.1 架构调整

```
┌─────────────────────────────────────────────────────────┐
│                      分类器                              │
│  ┌─────────────┐      ┌─────────────┐                   │
│  │  规则匹配    │─失败─▶│  VLM分类    │                   │
│  │ (微秒级)    │      │ (秒级)      │                   │
│  └─────────────┘      └─────────────┘                   │
│         │                    │                           │
│         ▼                    ▼                           │
│    确定的类型           确定的类型                        │
└─────────┬────────────────────┬──────────────────────────┘
          │                    │
          ▼                    ▼
┌─────────────────────────────────────────────────────────┐
│                      路由层                              │
│   ID_CARD ──▶ RULE    |  PURCHASE_CONTRACT ──▶ RULE     │
│   INVOICE ──▶ RULE    |  UNKNOWN ──▶ VLM                │
│   ...       ──▶ ...   |  (真正的未知，无法分类)          │
└─────────────────────────────────────────────────────────┘
```

### 5.2 方案优势

1. **准确率最高**：VLM处理所有边界情况
2. **职责清晰**：分类和提取分离
3. **路由灵活**：VLM分类后，仍可选择RULE层提取
4. **可维护性好**：规则层持续优化，VLM兜底保证下限

### 5.3 方案代价

1. 需要改造分类器（集成VLM调用）
2. 分类阶段增加延迟（约1-2秒/张）
3. VLM调用量增加（但从0→约20%，不是量级变化）

### 阶段3：精细化调优 ✅（2026-06-28 后续）

#### 修复1：命名对齐
- `DocumentType.PROPERTY_CERTIFICATE` 的值从 `"房产证"` 改为 `"不动产权证书"`（2015年起的官方标准名称）
- 消除了基线标注与代码之间的7个"假错误"

#### 修复2：多文档冲突检测
- 新增"阶段0"：当页面同时包含**买受人+出卖人+房屋类型**时，优先分类为购房/存量房合同
- 防止多文档混合扫描时，合同页面中的身份证号码误触发身份证分类
- 关键：仅用"买受人/出卖人"（合同专用术语），不用"甲方/乙方"（资金监管协议中也出现）

#### 发现：VLM对附属页面的判断是正确的
- 5张标注为"不动产权证书"但page_status="附属页面"的图片
- VLM将其中4张正确识别为"附属页面"
- 这说明VLM的视觉理解能力优于纯文本分析
