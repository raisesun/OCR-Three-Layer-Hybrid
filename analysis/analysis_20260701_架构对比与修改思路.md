# 当前架构 vs 技术方案对比与修改思路

**日期**: 2026-07-01  
**分析目标**: 对比现有系统与技术方案，提出架构修改建议

---

## 一、当前架构概览

### 1.1 技术栈

```
当前系统（OCR-Three-Layer-Hybrid）
├── OCR层: GLM-OCR (Q8_0量化，llama-server)
├── 分类层: 规则分类器 + VLM兜底
├── 提取层: 
│   ├── 规则层（RuleLayer）- 正则表达式
│   ├── VLM层（VLMLayer）- Qwen2.5-7B
│   └── LLM层（LLMLayer）- Qwen3.5-4B
├── 位置标注: PaddleOCR坐标提取（已实现）
└── 字段校验: 字段验证器（已实现）
```

### 1.2 实测表现

| 指标 | 数值 | 备注 |
|------|------|------|
| **分类准确率** | 66% (33/50) | 修正标注错误前 |
| **OCR速度** | 27秒/张 | GLM-OCR |
| **总耗时** | 62.9秒/张 | OCR+分类+提取 |
| **内存占用** | ~3GB | GLM-OCR + VLM |

### 1.3 核心问题

1. **OCR速度慢**: GLM-OCR 27秒/张，远高于预期
2. **分类准确率低**: 66%远低于目标90%+
3. **大量"未知"分类**: 分类器无法识别多种文档类型
4. **标注错误**: 基线文件存在标注不一致问题

## 二、技术方案架构

### 2.1 设计架构

```
技术方案（PP-StructureV3 + PaddleOCR-VL）
├── 预处理层: 方向矫正 + 去畸变 + 增强
├── OCR层: 双引擎按需调度
│   ├── PP-StructureV3 (A/B级文档) - 0.5-2秒/张
│   └── PaddleOCR-VL (C/D级文档) - 5-8秒/张
├── 分类层: 关键词匹配 + 引擎路由
├── 提取层:
│   ├── 规则解析器 (A/B级)
│   └── LLM提取器 (C/D级) - Qwen2.5:1.5B
└── 校验层: 格式校验 + 置信度评估
```

### 2.2 预期表现

| 指标 | 目标值 | 备注 |
|------|--------|------|
| **A级文档精度** | 99% | 身份证/结婚证等 |
| **B级文档精度** | 95% | 发票/户口本等 |
| **C级文档精度** | 90% | 合同/协议等 |
| **D级文档精度** | 80% | 病历/处方等 |
| **A级速度** | 0.5-2秒/张 | PP-StructureV3 |
| **C/D级速度** | 5-8秒/张 | PaddleOCR-VL |

## 三、架构差异分析

### 3.1 关键差异对比

| 维度 | 当前架构 | 技术方案 | 差异分析 |
|------|----------|----------|----------|
| **OCR引擎** | GLM-OCR (llama.cpp) | PP-StructureV3 + PaddleOCR-VL | 完全不同 |
| **OCR速度** | 27秒/张 | 0.5-8秒/张 | 当前慢3-50倍 |
| **分类方式** | VLM兜底 | 关键词匹配 | 当前更复杂 |
| **提取逻辑** | 规则+VLM+LLM三层 | 规则+LLM两层 | 当前多一层VLM |
| **校验机制** | 简单字段校验 | 格式校验+置信度 | 技术方案更完善 |
| **引擎数量** | 单引擎(GLM-OCR) | 双引擎 | 技术方案更灵活 |

### 3.2 问题根因分析

#### 问题1: OCR速度慢（27秒/张）

**根因**:
- GLM-OCR使用llama.cpp推理，CPU上效率低
- 模型过大（Q8_0量化后仍很大）
- 没有充分利用PaddleOCR的优化

**对比**:
- PP-StructureV3: 0.5-2秒/张（ optimized for CPU）
- PaddleOCR-VL: 5-8秒/张（VLM端到端）
- GLM-OCR: 27秒/张（llama.cpp通用推理）

**结论**: GLM-OCR不是为OCR任务优化的，应该替换

---

#### 问题2: 分类准确率低（66%）

**根因**:
1. **OCR文本质量差**: GLM-OCR识别错误多，导致分类器无法正确判断
2. **分类逻辑复杂**: VLM兜底增加了不确定性
3. **测试数据标注错误**: 基线文件存在标注不一致

**对比**:
- 技术方案: 关键词匹配分类（简单+快速）
- 当前方案: VLM分类（复杂+慢+不稳定）

**结论**: 分类器本身没问题，问题在于OCR质量

---

#### 问题3: 大量"未知"分类

**根因**:
- OCR文本太短或错误太多，分类器无法匹配
- 规则分类器的关键词覆盖不全
- VLM兜底失败（VLM服务不稳定）

**对比**:
- 技术方案: 先用PP-StructureV3快速OCR，再关键词分类
- 当前方案: GLM-OCR慢且不准，分类器无法工作

**结论**: 需要先解决OCR质量问题

---

### 3.3 核心结论

**当前系统的主要问题不是架构设计，而是OCR引擎选择错误**。

GLM-OCR是一个通用的视觉语言模型，不是专门为OCR任务优化的。它的特点是：
- ✅ 理解能力强（可以回答问题）
- ❌ OCR速度慢（27秒/张）
- ❌ 不适合批量OCR任务

而技术方案推荐的PP-StructureV3和PaddleOCR-VL是专门为OCR任务优化的：
- ✅ 速度快（0.5-8秒/张）
- ✅ 精度高（95%+）
- ✅ 针对文档场景优化

## 四、架构修改思路

### 4.1 修改原则

1. **保留现有架构的优点**:
   - 规则层+VLM层+LLM层的三层提取逻辑
   - 位置标注提取（已实现）
   - 字段校验（已实现）

2. **替换OCR引擎**:
   - GLM-OCR → PP-StructureV3 + PaddleOCR-VL
   - 保留VLM分类兜底（但使用更快的VLM）

3. **简化分类逻辑**:
   - 先用关键词匹配快速分类
   - 失败时再用VLM兜底

### 4.2 具体修改方案

#### 修改1: 替换OCR引擎

**当前**:
```python
# service.py
def run_ocr(self, image_path: str) -> str:
    # 调用GLM-OCR（llama-server）
    response = requests.post("http://localhost:8080/v1/chat/completions", ...)
    return response['choices'][0]['message']['content']
```

**修改后**:
```python
# service.py
def run_ocr(self, image_path: str, doc_type: str = None) -> str:
    # 根据文档类型选择OCR引擎
    if doc_type in ['身份证', '结婚证', '户口本', '发票']:
        # A/B级文档：使用PP-StructureV3（快）
        return self.structure_v3_ocr(image_path)
    else:
        # C/D级文档：使用PaddleOCR-VL（精度高）
        return self.paddleocr_vl_ocr(image_path)

def structure_v3_ocr(self, image_path: str) -> str:
    # PP-StructureV3推理
    from paddlex import PPStructureV3
    pipeline = PPStructureV3()
    result = pipeline.predict(image_path)
    return result['text']

def paddleocr_vl_ocr(self, image_path: str) -> str:
    # PaddleOCR-VL推理
    from paddlex import PaddleOCRVL
    pipeline = PaddleOCRVL()
    result = pipeline.predict(image_path)
    return result['text']
```

**预期效果**:
- OCR速度: 27秒 → 0.5-8秒（提升3-50倍）
- OCR精度: 提升（PaddleOCR专门优化）

---

#### 修改2: 简化分类逻辑

**当前**:
```python
# classifier.py
def classify(self, ocr_text: str) -> dict:
    # 1. 规则分类
    result = self.keyword_classifier.classify(ocr_text)
    
    # 2. 如果置信度低，使用VLM兜底
    if result['confidence'] < 0.8:
        result = self.vlm_classifier.classify(ocr_text, image_path)
    
    return result
```

**修改后**:
```python
# classifier.py
def classify(self, ocr_text: str, image_path: str = None) -> dict:
    # 1. 规则分类（快速）
    result = self.keyword_classifier.classify(ocr_text)
    
    # 2. 如果置信度低，使用VLM兜底（但使用更快的VLM）
    if result['confidence'] < 0.8 and image_path:
        # 使用Qwen2.5-7B（比GLM-OCR快）
        result = self.fast_vlm_classifier.classify(image_path)
    
    return result
```

**预期效果**:
- 分类速度: 提升（减少VLM调用）
- 分类准确率: 提升（OCR质量提升后）

---

#### 修改3: 优化VLM层

**当前**:
- VLM层使用Qwen2.5-7B（7B参数）
- 用于字段提取（复杂任务）

**修改后**:
- 保留Qwen2.5-7B用于复杂字段提取
- 新增Qwen2.5-1.5B用于简单分类任务（快）

**预期效果**:
- 分类速度: 提升（小模型更快）
- 提取精度: 保持（大模型更准）

## 七、总结

### 7.1 核心发现

1. **当前系统的主要问题不是架构，而是OCR引擎选择错误**
2. **GLM-OCR不适合批量OCR任务，应该替换**
3. **现有架构的优点（三层提取、位置标注、字段校验）应该保留**
4. **技术方案的双引擎设计合理，但需要实测验证**

### 7.2 修改思路

1. **替换OCR引擎**: GLM-OCR → PP-StructureV3 + PaddleOCR-VL
2. **简化分类逻辑**: 关键词匹配优先，VLM兜底
3. **优化VLM层**: 大模型提取，小模型分类
4. **分阶段实施**: 先替换OCR，再优化其他模块

### 7.3 预期效果

- OCR速度: 提升3-50倍
- 分类准确率: 从66%提升到90%+
- 总耗时: 从62.9秒降低到5-15秒
- A级文档精度: 达到99%
