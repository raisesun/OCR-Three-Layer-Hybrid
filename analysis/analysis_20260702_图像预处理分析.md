# 图像预处理分析报告

**日期**: 2026-07-02  
**目的**: 分析OCR三层混合架构是否需要图像预处理，参考AI-OCR项目实现

## 1. 问题背景

渐进式评测显示综合准确率75.6%，其中：
- 类型准确率: 90.0% ✅
- 字段准确率: 61.1% ❌

主要问题：
1. **规则层提取问题**: 8个字段错误（身份证2 + 发票2 + 户口本4）
2. **VLM识别问题**: 3个字段错误（购房合同形近字识别）
3. **分类问题**: 3个字段错误（合同页面缺少买受人）

## 2. AI-OCR预处理方案

### 2.1 预处理流水线

```
输入图片 → 去噪 → 纠偏 → 对比度增强 → [二值化] → OCR识别
```

### 2.2 各步骤分析

| 步骤 | 技术 | 目的 | 耗时估计 | 适用场景 |
|------|------|------|----------|----------|
| **去噪** | `cv2.fastNlMeansDenoisingColored` | 减少图像噪声，保留文字边缘 | ~200ms | 低质量扫描件、手机拍照 |
| **纠偏** | OTSU阈值 + minAreaRect + 旋转 | 修正倾斜文档 | ~300ms | 拍照文档、扫描倾斜 |
| **对比度增强** | CLAHE (自适应直方图均衡) | 增强文字与背景对比度 | ~100ms | 光照不均、褪色文档 |
| **二值化** (可选) | 自适应阈值/OTSU | 转为黑白二值图 | ~50ms | 减少颜色干扰 |

**总耗时**: ~650ms/张图片（不含二值化）

### 2.3 代码实现

```python
# AI-OCR的ImageEnhancer核心代码
class ImageEnhancer:
    def enhance(self, image: np.ndarray) -> np.ndarray:
        result = image.copy()
        
        # Step 1: 去噪
        if self.enable_denoise:
            result = cv2.fastNlMeansDenoisingColored(result, None, 10, 10, 7, 21)
        
        # Step 2: 纠偏
        if self.enable_deskew:
            result = self.deskew(result)  # OTSU + minAreaRect + 旋转
        
        # Step 3: 对比度增强
        result = self.enhance_contrast(result)  # CLAHE
        
        # Step 4: 二值化（可选）
        if self.enable_binarize:
            result = self.binarize(result, method="adaptive")
        
        return result
```

## 3. 当前项目分析

### 3.1 当前OCR流程

```
输入图片 → PP-OCRv6 (检测+方向分类+识别) → 文本后处理 → 规则层/VLM层提取
```

PP-OCRv6自带功能：
- **文档方向分类**: PP-LCNet_x1_0_doc_ori（自动旋转0°/90°/180°/270°）
- **文本行方向分类**: PP-LCNet_x1_0_textline_ori
- **UVDoc**: 文档扭曲矫正（针对曲面文档）

### 3.2 样本图片特征

从评测日志看：
```
Resized image size (3072x4096) exceeds max_side_limit of 4000.
```

说明输入图片分辨率较高（3072x4096），PP-OCRv6会自动缩放到4000以内。

### 3.3 错误根因分析

| 错误类型 | 根因 | 预处理能否解决 |
|----------|------|----------------|
| 身份证"值+标签"格式 | OCR输出顺序问题 | ❌ 不能，需要规则层修复 |
| 发票价税合计括号格式 | OCR输出半角/全角不一致 | ❌ 不能，需要规则层修复 |
| 户口本"登记事项变更"页格式 | 特殊表格布局导致OCR分行 | ❌ 不能，需要规则层修复 |
| 购房合同"张玉煊"→"张玉霞" | VLM形近字识别错误 | ⚠️ 可能有帮助，但需验证 |

**结论**: 当前错误主要是**格式适配问题**，而非图像质量问题。

## 4. 方案对比

### 方案A: 不添加预处理（当前状态）

**优点**:
- 简单，无额外依赖
- 处理速度快（无预处理耗时）
- PP-OCRv6已内置方向矫正

**缺点**:
- 无法处理低质量图片
- VLM识别错误无法改善

### 方案B: 添加轻量级预处理

**预处理内容**:
- 对比度增强（CLAHE）
- 可选去噪

**优点**:
- 耗时增加少（~300ms/张）
- 可能改善VLM识别
- 对低质量图片有帮助

**缺点**:
- 对当前错误帮助有限（主要是格式问题）
- 增加OpenCV依赖

### 方案C: 添加完整预处理流水线

**预处理内容**:
- 去噪 + 纠偏 + 对比度增强 + 二值化

**优点**:
- 全面提升图像质量
- 对各类低质量文档都有帮助

**缺点**:
- 耗时增加明显（~650ms/张）
- 可能引入新问题（如纠偏错误）
- 二值化丢失灰度信息，影响VLM

### 方案D: 按需预处理（推荐）

**策略**:
1. 默认不预处理（依赖PP-OCRv6内置能力）
2. 当识别置信度低时，触发预处理重试
3. 预处理内容：仅对比度增强（CLAHE）

**优点**:
- 平衡速度和精度
- 针对性处理低质量图片
- 避免过度处理

**缺点**:
- 实现复杂度高
- 需要定义"置信度低"的阈值

## 5. 推荐方案

**短期（当前优先级）**: 方案A（不添加预处理）

**理由**:
1. 当前错误主要是**格式适配问题**，预处理无法解决
2. 规则层修复后，准确率应能显著提升
3. 先验证规则层修复效果，再评估是否需要预处理

**中期（待评估）**: 方案B（轻量级预处理）

**触发条件**:
- 规则层修复后，仍有VLM识别错误
- 遇到低质量图片（模糊、光照不均）

**实施建议**:
1. 在`OCRService.run_ocr()`前添加可选的`preprocess()`方法
2. 默认关闭，通过配置启用
3. 仅实现CLAHE对比度增强（最简单、最安全）
