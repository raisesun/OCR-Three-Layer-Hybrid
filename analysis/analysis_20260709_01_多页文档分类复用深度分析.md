## 1. 原始需求

> 多页文档，在不同的页面上会有我们需要的数据项目的一部分或全部，并且有一些页面不需要获取数据项目。因此处理一个图片时要知道它的文档分类和细分类型，根据细分类型来确定从图片上获取哪些数据项目或者不获取数据项目，然后将这个子业务中同一个文档分类的图片中获取的数据项目合并到一起形成这个文档分类的数据结果。

**需求拆解**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 逐页独立分类 | 每页得到 `doc_type` + `page_type` → 细化为 `refined_doc_type` |
| 2 | 根据细分类型决定提取策略 | COVER/STAMP → 跳过；CONTENT → 提取特定字段 |
| 3 | 按细分类型提取字段 | 每种细分类型有独立的 `key_list` 和 `prompt` |
| 4 | 合并同一文档的结果 | 取各页提取字段的第一个非空值，冲突记录 |

## 3. 发现的全部问题

### 3.1 核心 Bug：_extract_multi_page_merge 复用首页分类

```python
# service.py:414-545
def _extract_multi_page_merge(self, image_paths, doc_info):
    for page_idx, img_path in enumerate(image_paths):
        ...
        result = self._pipeline.process(img_path, ocr_texts, doc_info=doc_info)
        #                                                           ↑↑↑
        #                            所有页都传入同一个 doc_info（来自第1页）
```

**后果**：
- 第1页分类为 `HOUSEHOLD_REGISTER_COVER` → 所有页都当封面处理
- 封面页的 key_list 为空 → 所有页提取 0 个字段
- 最终合并结果：全部为空

### 3.2 key_lists 大量缺失

| 细分类型 | key_list | 影响 |
|----------|----------|------|
| HOUSEHOLD_REGISTER_COVER | `[]` 空 | 即使分类正确也提取 0 字段 |
| DIVORCE_CERTIFICATE_CONTENT | `[]` 空 | ⚠️ 内容页无法提取！核心数据页 |
| PROPERTY_CERTIFICATE_CONTENT | `[]` 空 | 内容页无法提取 |
| PROPERTY_CERTIFICATE_ATTACHMENT | `[]` 空 | 附图页（本应跳过但用空 key_list 而非显式跳过） |
| MARRIAGE_CERTIFICATE_COVER | `[]` 空 | 封面（本应跳过） |
| MARRIAGE_CERTIFICATE_STAMP | `[]` 空 | 盖章页（本应跳过） |
| DIVORCE_CERTIFICATE_COVER/STAMP | `[]` 空 | 封面/盖章页 |
| PURCHASE_CONTRACT_STAMP | `[]` 空 | 签署页 |
| STOCK_CONTRACT_STAMP | `[]` 空 | 签署页 |
| FUND_SUPERVISION_AGREEMENT_STAMP | `[]` 空 | 签章页 |

**影响**：
- COVER/STAMP 页 key_list 为空 → 效果等同于跳过（可接受，但不够显式）
- **DIVORCE_CERTIFICATE_CONTENT key_list 为空 → 内容页无法提取数据 → 离婚证整本提取失败！**
- PROPERTY_CERTIFICATE_CONTENT key_list 为空 → 房产证内容页提取失败

**后果**：
- HOUSEHOLD_REGISTER_COVER 不跳过 → 会尝试用 `extract_household_register` 提取 → key_list 为空 → 返回空结果（功能上等效但浪费计算）
- PURCHASE_CONTRACT_STAMP 不跳过 → 走到 line 197-204 才返回空（逻辑正确但路径混乱）

### 3.4 路由逻辑类型不匹配

```python
# service.py:436-443
multi_page_types = {
    DocumentType.PURCHASE_CONTRACT,       # ← 基础类型
    DocumentType.STOCK_CONTRACT,
    DocumentType.PROPERTY_CERTIFICATE,
    DocumentType.UNKNOWN,
}

if doc_type in multi_page_types:   # ← doc_type 是细化后的类型！
    → _extract_multi_page_vlm
else:
    → _extract_multi_page_merge
```

`doc_type` 来自 `doc_info.doc_type`，是细化后的类型（如 `PURCHASE_CONTRACT_FIRST_PAGE`）。
但 `multi_page_types` 包含基础类型（`PURCHASE_CONTRACT`）。

**后果**：
- `PURCHASE_CONTRACT_FIRST_PAGE` ∉ `multi_page_types` → 错误地走 merge 路径
- 本应走 VLM 多页提取的合同，变成了逐页 merge（每页独立调 VLM，效率低）
- 只有恰好分类为 `PURCHASE_CONTRACT`（未细化）时才走正确路径

### 3.5 最终结果的 doc_type 使用首页细化类型


**后果**：返回结果的 doc_type 是封面类型，而非文档的基础类型。调用方无法正确识别这个结果属于哪个文档类型。

### 3.6 FUND_SUPERVISION 不在 multi_page_types 中

```python
# service.py:436-441
multi_page_types = {
    DocumentType.PURCHASE_CONTRACT,
    DocumentType.STOCK_CONTRACT,
    DocumentType.PROPERTY_CERTIFICATE,
    DocumentType.UNKNOWN,
    # 缺少 FUND_SUPERVISION

```

**后果**：资金监管协议（首页+信息页+签章页+凭证，典型多页文档）始终走 merge 路径，受 Bug #3.1 影响。

### 3.7 VLM _get_base_doc_type 映射不完整

vlm_layer.py 的 base_mapping 缺少以下映射：
- `PROPERTY_CERTIFICATE_FIRST_PAGE` → 不在 mapping 中
- `PURCHASE_CONTRACT_FIRST_PAGE/CONTENT/STAMP` → 不在 mapping 中
- `STOCK_CONTRACT_FIRST_PAGE/CONTENT/STAMP` → 不在 mapping 中
- `FUND_SUPERVISION_CERTIFICATE` → 不在 mapping 中（有独立 Prompt，不影响）

**后果**：回退逻辑仍能工作（因为基础类型有 prompt），但代码不完整，未来添加新类型时容易遗漏。

### 3.8 key_list 与 VLM prompt 字段不一致

以 `HOUSEHOLD_REGISTER_CONTENT` 为例：
- key_list: `["户主姓名", "户号", "住址", "姓名", "与户主关系", "性别", "公民身份号码"]`（7 字段）
- VLM prompt: 只定义了 `姓名, 与户主关系, 性别, 出生日期, 民族, 公民身份号码`（6 字段）
- 差异：key_list 有"户主姓名/户号/住址"，但 prompt 未要求 VLM 提取这些字段

**后果**：VLM 提取结果中缺少 key_list 要求的字段，merge 后这些字段为空。

### 5.2 方案 A：_extract_multi_page_merge 逐页独立分类

```python
def _extract_multi_page_merge(self, image_paths, first_page_doc_info):
    """逐页独立分类 + 提取 + 合并"""
    merged_fields = {}
    page_results = []

    for page_idx, img_path in enumerate(image_paths):
        ocr_text = self.run_ocr(img_path)
        ocr_texts = [ocr_text] if ocr_text else []

        # 首页复用已有分类，后续页独立分类
        if page_idx == 0:
            page_doc_info = first_page_doc_info
        else:
            page_doc_info = self._classifier.classify(img_path, ocr_texts)

        # 跳过封面页/盖章页（key_list 为空的细分类型）
        key_list = self._pipeline.key_lists.get(page_doc_info.doc_type, [])
        if not key_list:
            logger.info("[多页] 页 %d | %s | 跳过（无需提取的字段）",
                       page_idx, page_doc_info.doc_type.value)
            continue

        result = self._pipeline.process(img_path, ocr_texts, doc_info=page_doc_info)
        page_results.append((f"page_{page_idx}", page_doc_info.doc_type, result))

        # 合并字段（取第一个非空值）
        if result.success:
            for key, value in result.fields.items():
                if value and value.strip() and not merged_fields.get(key):
                    merged_fields[key] = value

    # 结果使用基础文档类型
    base_doc_type = self._get_base_doc_type(first_page_doc_info.doc_type)
    ...
```

## 9. 最终确认的架构方案（2026-07-09 第二轮讨论后确定）

### 9.0 核心原则：RULE 优先，VLM 兜底

**所有文档（单页/多页）统一策略**：先走 OCR + RULE 层提取，如果 RULE 层失败（必填字段未提取到），再对该页使用 VLM 兜底。

适用范围：所有证件（户口本、结婚证、离婚证、房产证）、所有合同（购房合同、存量房合同）、所有协议（资金监管协议、离婚协议）、所有单证（资金监管凭证）。

### 9.1 字段配置：独立 `field_configs.py`

每个文档类型/子类型必须明确定义：
- 提取哪些字段（FieldDef 列表）
- 每个字段是否必须提取到（`required=True/False`）
- 是否明确跳过不提取（`skip=True`）

```python
@dataclass
class FieldDef:
    name: str
    required: bool     # True=必须提取到，False=允许提取不到

@dataclass
class DocumentFieldConfig:
    doc_type: DocumentType
    fields: List[FieldDef]
    skip: bool = False  # True=明确定义不提取任何数据
```

#### 各文档类型的字段定义（初始版本，用户可直接修改 required 值）

| 文档类型 | required 字段 | optional 字段 |

### 9.2 RULE 层失败判定与日志

**失败判定**：任一 `required=True` 的字段值为空 → RULE 层失败 → 触发 VLM 兜底

**两种失败情况需在日志中区分**：

| 情况 | 含义 | 日志标记 |
|------|------|----------|
| `located_empty` | 正则/规则匹配到了字段位置，但值为空 | `[已定位空值]` |
| `not_found` | 正则/规则完全没有匹配到该字段 | `[未找到]` |

```python
@dataclass
class FieldDetail:
    name: str
    value: str               # 提取到的值
    status: str              # "extracted" | "located_empty" | "not_found"
```

日志示例：
```
[RULE提取] 页 2 | DIVORCE_CERTIFICATE_CONTENT
  [已提取] 离婚证字号 = "L340321-2018-002721"
  [未找到] 持证人身份证件号
  [已定位空值] 登记日期 = ""
  → RULE 结果: required 1/3 满足 → 失败
  → 触发 VLM 兜底: 缺失必填字段 [持证人身份证件号, 登记日期]
```

### 9.3 VLM 兜底策略（方案 C）

- **触发条件**：RULE 层失败（required 字段无值）
- **粒度**：仅对失败页做兜底，其他页不受影响
- **Prompt 构建**：VLM prompt 包含所有字段定义（保证 prompt 完整）
- **结果合并**：只用 VLM 的值填充 RULE 层缺失的字段，RULE 已成功的字段不覆盖
- **冲突处理**：RULE 已成功字段始终保留，VLM 值不同时记录日志

```
字段状态矩阵：

                  RULE 结果
                  ├─ 有值    ├─ 空（已定位）  ├─ 未找到
    ────────────────────────────────────────────────
VLM  │ 有值   │ 保留 RULE ✅  │ 用 VLM 填     │ 用 VLM 填
结果 │ 空值   │ 保留 RULE ✅  │ 记录日志 ⚠️    │ 记录日志 ⚠️
     │ 不一致 │ 记录日志 ⚠️   │ 用 VLM 填     │ 用 VLM 填
```

### 9.4 修改后的完整处理流程

```
process_multi_page([page1, page2, ..., pageN])
│
├─ 分类第1页 → doc_info
│
├─ FOR each page:
│   ├─ 独立分类 → page_doc_info.doc_type（细分类型）
│   │
│   ├─ 从 field_configs 获取该类型的配置:
│   │   config = field_configs[page_doc_info.doc_type]
│   │
│   ├─ IF config.skip == True:
│   │   日志: "页 {idx} | {type} | 跳过（明确定义不提取）"
│   │   CONTINUE
│   │
│   ├─ RULE 层提取:
│   │   result = rule_extractor.extract(page_doc_info, config.fields)
│   │   日志: 逐字段记录（已提取/已定位空值/未找到）
│   │
│   ├─ 检查 RULE 层结果:
│   │   required_missing = [f for f in config.fields
│   │                       if f.required and result[f.name] 为空]
│   │
│   ├─ IF required_missing 不为空:
│   │   日志: "RULE层失败 | 缺失必填字段: [...]"
│   │   → VLM 兜底:
│   │     vlm_result = vlm_layer.extract(page_doc_info, all_field_names)
│   │     合并: 用 VLM 结果填充 RULE 层缺失的字段
│   │     冲突: RULE 已有的字段保留，VLM 值不同则记录日志
│   │
│   └─ 合并到 merged_fields
│
└─ 返回: ExtractionResult(doc_type=base_type, fields=merged_fields)
```
