## 一、问题起点

### 1.1 初始现象

批量测试50张样本后，分类器准确率仅 **30% (15/50)**。

各文档类型准确率：

| 文档类型 | 正确 | 总数 | 准确率 |
|---------|------|------|--------|
| hukou | 2 | 12 | 17% |
| id_card_back | 3 | 5 | 60% |
| id_card_front | 0 | 10 | 0% |
| marriage | 5 | 8 | 62% |
| property | 3 | 7 | 43% |
| purchase_contract | 2 | 8 | 25% |

### 1.2 初始假设

初步认为是**关键词优先级问题**：户口本优先于结婚证导致户口本被误判为结婚证，购房合同被识别为身份证。

用户要求深入分析，找出比"修改提示词优先级"更加准确和稳定的解决方案。

## 二、第一轮：深度根因分析（苏格拉底提问法）

### 2.1 当前分类器的本质

```
输入: OCR文本列表 → 拼接为全文
处理: 对每种文档类型，检查是否存在关键词
输出: 匹配到多个类型时，按预定义优先级选最高的
```

**核心机制**: 关键词存在性检测 + 静态优先级裁决

### 2.2 基本假设验证

| 假设 | 内容 | 是否成立 | 原因 |
|------|------|---------|------|
| 假设1 | 关键词能唯一标识文档类型 | ❌ | "身份证"出现在6种文档中 |
| 假设2 | 文档出现关键词=文档属于该类型 | ❌ | 结婚证备注含"身份证件号"，不等于这是身份证 |
| 假设3 | 静态优先级能正确裁决冲突 | ❌ | 强信号"商品房买卖合同"优先级6(最低)，弱信号"身份证"优先级1(最高) |

### 2.3 苏格拉底追问链

**Q1: 为什么户口本被误判为结婚证？(9/12 = 75%错误率)**

A1: 户口本和结婚证同时匹配了关键词，结婚证优先级(2)高于户口本(3)。

**Q2: 户口本匹配了哪些关键词？结婚证匹配了哪些？**

A2:
- 户口本匹配了："户主"（页面有"户主姓名"字段）
- 结婚证匹配了："登记日期"（页面有"登记事项变更和更正记载"表头）
- 有些还匹配了："公民身份号码"（个人页有身份证号）

**Q3: 为什么"登记日期"会出现在户口本中？**

A3: 户口本的"登记事项变更和更正记载"页面是表格，表头有"登记日期"列名。这是行政表格通用格式，不是结婚登记特有的。

**Q4: 户口本最独特的标识是什么？**

A4: **"常住人口登记卡"** — 户口本个人页的标题，其他任何文档都不会出现这个词。但当前规则中**完全没有这个关键词**！

**Q5: 为什么没有"常住人口登记卡"这个关键词？**

A5: 规则设计者使用了泛化的词("户口本"、"户口簿")，但实际文档中的措辞是更正式的"常住人口登记卡"。这是关键词设计与实际文档用语不匹配的问题。

**Q6: 所以根因是什么？**

A6:
1. **缺失强特征关键词**："常住人口登记卡"是户口本最强信号，但未被纳入
2. **弱关键词冲突**："登记日期"在多种文档中出现，不是好的区分特征
3. **优先级方向错误**：户口本的优先级(3)低于结婚证(2)，但"户主"比"登记日期"更具区分力

**Q7: 为什么结婚证被误判为身份证？(3/8 = 37.5%错误率)**

A7: 结婚证备注区域包含"身份证件号"字段，触发了"身份证"关键词。

**Q8: 为什么"身份证"关键词会匹配到"身份证件号"？**

A8: 因为代码使用 `keyword in full_text` 子串匹配。"身份证"是"身份证件号"的子串。

**Q9: "身份证件号"和"公民身份号码"有什么区别？**

A9:
- "公民身份号码" → 出现在**身份证本体**上，是身份证的强特征
- "身份证件号" → 出现在**其他证件**的备注中，引用当事人的身份证号
- 前者说明"这是身份证"，后者说明"这不是身份证，但提到了身份证"

**Q10: 所以根因是什么？**

A10:
1. **关键词粒度过粗**："身份证"太短，会误匹配"身份证件号"、"身份证复印件"等
2. **缺少上下文区分**：无法区分"这是身份证本体"和"其他文档引用了身份证号"
3. **优先级过高**：ID_CARD优先级为1，一旦匹配就碾压其他类型

**Q11: 为什么购房合同被误判为身份证？(2/8 = 25%错误率)**

A11: 合同条款中提到"买受人应提供居民身份证复印件"等文字。

**Q12: 合同中明明有"商品房买卖合同"这个强特征词，为什么没生效？**

A12: 因为"商品房买卖合同"匹配了PURCHASE_CONTRACT(优先级6)，但"身份证"匹配了ID_CARD(优先级1)，优先级1碾压了优先级6。

**Q13: 这说明了什么？**

A13: **当前优先级体系完全颠倒了！** 应该是：
- "商品房买卖合同" → 强信号，几乎100%确定是购房合同
- "身份证"出现在合同文本中 → 弱信号，不能说明这是身份证

但系统给了弱信号更高的优先级。

### 2.4 根因总结

```
问题表现: 分类器准确率仅30%
│
├── 根因1: 关键词不是唯一标识符 (关键词重叠)
│   ├── "身份证" 出现在 6种文档中
│   ├── "登记日期" 出现在 4种文档中
│   └── "户主" 同时出现在户口本和结婚证的上下文中
│
├── 根因2: 子串匹配产生假阳性
│   ├── "身份证" 匹配 "身份证件号"(其他文档引用)
│   ├── "登记日期" 匹配 "抵押登记日期"(合同条款)
│   └── "存量房" 匹配 "存量房交易资金监管"(监管协议)
│
── 根因3: 优先级体系有根本缺陷
│   ├── 强信号("商品房买卖合同")优先级6(最低)
│   ├── 弱信号("身份证")优先级1(最高)
│   ── 优先级是静态的，无法根据上下文调整
│
├── 根因4: 关键强特征词缺失
│   ├── 户口本缺少"常住人口登记卡"
│   ├── 房产证缺少"不动产权证书 BDCQZ"
│   └── 合同缺少"GF-2014-0171"等标准编号
│
└── 根因5: 部分测试样本标签可能有误
    └── case文件夹含多种文档，页面类型标错
```

## 三、第二轮：解决方案对比

### 3.1 五个方案对比

| 方案 | 做法 | 优点 | 缺点 | 预期准确率 |
|------|------|------|------|----------|
| A: 修复关键词+调优先级 | 添加缺失关键词，调整优先级 | 改动最小，立即可做 | 治标不治本 | 30%→50-60% |
| B: 强/弱关键词分级 | 强信号直接判定，弱信号参与评分 | 解决强弱混淆 | 需仔细定义强弱等级 | 30%→70-80% |
| C: 多信号加权评分 | 每个关键词有权重，计算加权得分 | 最灵活 | 权重需训练/调参 | 30%→80-90% |
| D: 两阶段分类（推荐） | 阶段1强信号快速判定，阶段2弱信号评分，阶段3 VLM兜底 | 兼顾速度和准确性 | 实现复杂度中等 | 30%→85-95% |
| E: 使用VLM分类 | 完全跳过关键词，用VLM判断类型 | 最准确 | 慢(10-30s/张)，成本高 | 95%+ |

### 3.2 初步推荐

推荐方案D（两阶段分类）为主，融合方案B（强弱关键词分级）。

### 3.3 强信号覆盖率分析

用户追问"有哪些文档没有强信号？"，进行了覆盖分析：

| 文档类型 | 有强信号 | 无强信号 | 覆盖率 | 无强信号原因 |
|---------|---------|---------|--------|-------------|
| id_card_back | 5/5 | 0 | **100%** | ✅ 全部有"签发机关" |
| hukou | 11/12 | 1 | **92%** | 1张是封面页（无"常住人口登记卡"） |
| id_card_front | 9/10 | 1 | **90%** | 1张实际是离婚协议书（标签错误） |
| property | 5/7 | 2 | **71%** | 宗地图、附图页（无"不动产权证书"） |
| marriage | 5/8 | 3 | **63%** | 结婚证内页（有"结婚"但无"结婚证字号"） |
| purchase_contract | 4/8 | 4 | **50%** | ️ 合同中间页（无"商品房买卖合同"标题） |

**关键发现**：
- 购房合同50%页面没有强信号（中间页无标题）
- 结婚证37.5%页面没有强信号（内页无字号）
- 强信号只能覆盖 ~65-70% 的案例

## 四、第三轮：业务理解修正

### 4.1 初始错误理解

最初认为OCR系统的目的是**自动填充税务系统表单**。

### 4.2 用户纠正

用户指出：
> "我们不是提取数据自动填充系统，我们的目的是使用图片中的原始单证证明业务真实性，同时要从图片中的原始单证中提取出数据与系统记录数据进行比对，发现双方的数据差异，如果有差异要分析原因、判断是缺失了数据还是填错了内容或者是业务真实性存疑。"

### 4.3 正确业务理解

```
目标: 用原始单证验证税务系统数据的真实性
流程:
1. 从图片提取原始单证数据
2. 与系统记录数据比对
3. 发现差异 → 分析原因（漏填？填错？造假？）
4. 单证之间交叉验证 → 发现内在矛盾
```

**分类器的真正目标**：将每个文档路由到正确的"提取+验证规则集"

## 五、第四轮：商品房 vs 存量房 验证重点差异

### 5.1 业务流程差异

| 环节 | 商品房（新房） | 存量房（二手房） |
|-----|-------------|---------------|
| 初审 | ❌ 无初审，直接复审 | ✅ 有人工初审（查家庭住房情况表） |
| 评估 | ❌ 不需要评估，按纳税人填写金额计税 | ✅ 需要评估核价（系统自动评估+人工核价） |
| 复审 | ✅ 确认信息+开发商信息 | ✅ 确认信息+评估价格 |

### 5.2 需要验证的文档差异

| 文档类型 | 商品房 | 存量房 | 验证目的 |
|---------|-------|-------|---------|
| 商品房买卖合同 | ✅ 核心 | ❌ | 合同价格、买卖双方、房屋信息 |
| 存量房买卖合同 | ❌ | ✅ 核心 | 合同价格、买卖双方、房屋信息 |
| 不动产权证书 | ✅ 有 | ✅ 有 | 产权人、面积、单元号 |
| 身份证（买方） | ✅ | ✅ | 买方身份 |
| 身份证（卖方） | ❌ 卖方是开发商 | ✅ | 卖方身份 |
| 结婚证 | ✅ 有 | ✅ 有 | 婚姻状况→免税优惠资格 |
| 户口本 | ❓ 可能 | ✅ 有 | 家庭住房套次 |
| 增值税发票 | ✅ 有 | ❓ 可能 | 不含税金额、税额 |
| 资金监管协议 |  | ✅ 有 | 交易资金真实性 |

### 5.3 系统表单字段差异

#### 商品房特有字段：
- **开发商名称** / **开发商识别号**（从合同提取，验证开发商资质）
- **增值税税额** / **不含增值税金额**（从发票提取）
- **不动产项目名称**（从金三系统获取，与合同比对）

#### 存量房特有字段：
- **评估价格**（从评估系统获取，与合同价格比对→防止低报）
- **前次交易日期** / **前次权证填制日期**（从房产证提取→判断持有年限→影响增值税）
- **上手取得房屋方式**（买卖/继承/赠与→影响个税）
- **房改房抵扣总额**（特殊业务）
- **土地使用权出让金**（土地交易信息）

### 5.4 验证重点差异

| 验证维度 | 商品房重点 | 存量房重点 |
|---------|----------|----------|
| **价格真实性** | 合同价格 vs 发票金额 | 合同价格 vs 评估价格（核心！防止低报逃税） |
| **身份验证** | 买方身份证 | 买卖双方身份证 |
| **产权验证** | 开发商资质+项目登记 | 房产证权利人=卖方 |
| **税收优惠** | 家庭住房套次（契税减免） | 婚姻状况+家庭住房套次+持有年限 |
| **特殊业务** | 开发商识别号验证 | 房改房、继承、赠与等特殊取得方式 |

### 5.5 交叉验证规则建议

#### 身份证 ↔ 所有文档（身份一致性）

| 验证项 | 规则 | 异常含义 |
|-------|------|---------|
| 身份证号 | 身份证、户口本个人页、结婚证备注、购房合同买受人/出卖人 → 应一致 | 身份造假/张冠李戴 |
| 姓名 | 同上所有文档的姓名应一致 | 同上 |

#### 结婚证 ↔ 户口本（婚姻一致性）

| 验证项 | 规则 | 异常含义 |
|-------|------|---------|
| 婚姻状况 | 户口本"婚姻"栏=已婚 ↔ 有结婚证 | 隐瞒婚姻状态（影响免税） |
| 配偶信息 | 结婚证双方姓名/身份证号 ↔ 户口本配偶栏 | 配偶信息不一致 |
| 登记日期 | 结婚证登记日期 < 交易日期 | 交易时未婚但声称已婚（骗税） |

#### 房产证 ↔ 购房合同（产权一致性）

| 验证项 | 规则 | 异常含义 |
|-------|------|---------|
| 房屋地址 | 房产证地址 ≈ 合同房屋坐落 | 合同与产权不匹配 |
| 权利人/买受人 | 房产证权利人 = 合同买受人 | 产权人与买方不一致 |
| 面积 | 房产证建筑面积 ≈ 合同面积 | 面积差异（可能逃税） |
| 不动产单元号 | 合同中有 → 应与房产证一致 | 单元号不匹配 |

#### 购房合同内部（价格逻辑）

| 验证项 | 规则 | 异常含义 |
|-------|------|---------|
| 总价 = 单价 × 面积 | 合同内数字关系 | 价格计算错误/篡改 |
| 合同价格 vs 系统交易价格 | 应一致 | 低报价格逃税 |
| 签订日期 | 合同日期 < 交易日期 | 日期逻辑错误 |

#### 资金监管协议（如果有）↔ 合同

| 验证项 | 规则 | 异常含义 |
|-------|------|---------|
| 交易金额 | 监管金额 = 合同价格 | 资金流向与合同不符 |
| 买卖双方 | 监管协议双方 = 合同双方 | 交易主体不一致 |

### 5.6 验证链设计

#### 商品房验证链：
```
商品房买卖合同 → 提取：合同价格、买受人、房屋地址
       ↓
增值税发票 → 提取：不含税金额、税额
       ↓
交叉验证：合同价格 = 不含税金额 + 税额
       ↓
身份证 → 提取：买方姓名、身份证号
       ↓
结婚证/户口本 → 提取：婚姻状况、家庭套次
       ↓
系统比对：合同价格 vs 系统录入价格
         买受人 vs 系统买方
         家庭套次 vs 系统申报套次
```

#### 存量房验证链：
```
存量房买卖合同 → 提取：合同价格、买卖双方、房屋地址
       ↓
房产证 → 提取：权利人、面积、单元号、前次交易日期
       ↓
交叉验证：合同买方 = 房产证权利人
         合同面积 ≈ 房产证面积
         前次交易日期 → 计算持有年限
       ↓
资金监管协议 → 提取：监管金额、买卖双方
       ↓
交叉验证：监管金额 = 合同价格
         监管双方 = 合同双方
       ↓
身份证（双方）→ 提取：姓名、身份证号
       ↓
结婚证/户口本 → 提取：婚姻状况
       ↓
系统比对：合同价格 vs 评估价格（关键！）
         合同双方 vs 系统交易双方
         持有年限 vs 系统前次交易日期
```

## 六、对分类器设计的最终影响

### 6.1 设计原则转变

| 从 | 到 |
|---|---|
| "这是哪种文档类型" | "这个文档能提供哪些验证字段" |
| 强信号→类型判定 | 强信号→路由到正确的提取+验证规则集 |
| UNKNOWN→放弃 | UNKNOWN→VLM判断"这个页面有什么可提取的验证信息" |

### 6.2 修订后的方案D设计

```
输入: OCR文本
↓
阶段1: 强信号判定（能路由到哪个提取器）
  - "常住人口登记卡" → 户口本提取器
  - "结婚证字号" → 结婚证提取器
  - "公民身份号码" → 身份证提取器
  - "商品房买卖合同"或"存量房买卖合同" → 合同提取器
  - "不动产权证书" → 房产证提取器
↓
阶段2: 弱信号评分（当强信号缺失时）
  - 合同中间页：如果有"买受人"+"出卖人"+"价款" → 路由到合同提取器
  - 房产证附图：如果有"宗地代码"+"权利人" → 路由到房产证提取器
↓
阶段3: VLM兜底（无法路由时）
  - 让VLM判断"这个页面能提供什么信息"
```

### 6.3 关键变化

1. **不强求每个页面都有明确的文档类型**，而是判断"这个页面是否有系统需要的字段"
2. **新文档类型（资金监管协议、存量房买卖合同）**不应该被丢弃，它们包含关键验证数据
3. **中间页的价值**：合同中间页如果包含价格信息，对存量房尤其重要（因为要与评估价比对）

### 9.2 苏格拉底追问：为什么需要区分这两类文档？

**Q1: 身份证、结婚证、房产证、户口本有什么共同特点？**

A1: 它们是**标准证件**，有固定格式和强特征词：
- 身份证："公民身份号码"、"签发机关"
- 结婚证："结婚证字号"
- 房产证："不动产权证书 BDCQZ"
- 户口本："常住人口登记卡"

这些词**几乎只出现在对应证件上**，误判率极低。

---

**Q2: 买卖合同、资金监管协议、离婚协议有什么共同特点？**

A2: 它们是**长文本合同/协议**，特点：
- 格式不固定（不同地区、不同时期格式不同）
- 标题可能不在拍摄的页面中（中间页）
- 内容重叠（都可能出现"买方"、"卖方"、"金额"）
- 新类型可能出现（如"存量房买卖合同"不在原分类表中）

---

### 9.3 核心问题：是否需要精确分类？

**Q3: 系统真正需要的是什么——文档类型标签，还是验证字段？**

A3: 系统需要的是**验证字段**，不是类型标签。

| 系统需要的 | 来源文档 |
|----------|---------|
| 合同价格 | 买卖合同（任何一页，只要包含价格条款） |
| 买卖双方姓名 | 买卖合同、资金监管协议 |
| 监管金额 | 资金监管协议 |
| 财产分割信息 | 离婚协议（可能不需要？） |

---

**Q4: 如果只需要字段，能否用"字段提取路由"代替"文档类型路由"？**

A4: 可以！设计思路：

```
传统路由: 文档 → 类型判定 → 类型专属提取器
新路由:   文档 → 字段检测 → 字段专属提取器
```

### 9.6 修订后的路由逻辑代码

```python
def classify_document(ocr_text: str) -> str:
    """
    三类路由逻辑：
    1. 标准证件强信号 → 直接路由
    2. 标准单证强信号 → 直接路由
    3. 合同/协议字段组合 → 路由到对应提取器
    4. 无法判定 → VLM兜底
    """
    
    # === 阶段1: 标准证件强信号 ===
    if "公民身份号码" in ocr_text or "签发机关" in ocr_text:
        return "ID_CARD"
    if "结婚证字号" in ocr_text:
        return "MARRIAGE_CERTIFICATE"
    if "常住人口登记卡" in ocr_text:
        return "HOUSEHOLD_REGISTER"
    if "不动产权证书" in ocr_text:
        return "PROPERTY_CERTIFICATE"
    
    # === 阶段2: 标准单证强信号 ===
    # 发票特征
    if "发票代码" in ocr_text and "发票号码" in ocr_text:
        return "INVOICE"
    if "税额" in ocr_text and "不含税金额" in ocr_text:
        return "INVOICE_WEAK"  # 中置信度
    
    # === 阶段3: 合同/协议字段组合 ===
    # 买卖合同特征
    has_buyer = "买受人" in ocr_text or "买方" in ocr_text
    has_seller = "出卖人" in ocr_text or "卖方" in ocr_text
    has_price = "总价款" in ocr_text or "价款" in ocr_text or "合同金额" in ocr_text
    has_property_type = "商品房" in ocr_text or "存量房" in ocr_text
    
    if has_buyer and has_seller and has_price:
        if has_property_type:
            return "PURCHASE_CONTRACT"  # 高置信度
        else:
            return "PURCHASE_CONTRACT_WEAK"  # 中置信度
    
    # 资金监管协议特征
    has_supervision = "资金监管" in ocr_text or "监管协议" in ocr_text
    has_supervision_amount = "监管金额" in ocr_text or "监管价款" in ocr_text
    
    if has_supervision and has_supervision_amount:
        return "FUND_SUPERVISION"
    
    # 离婚协议特征
    has_divorce = "离婚" in ocr_text and ("财产分割" in ocr_text or "抚养" in ocr_text)
    if has_divorce:
        return "DIVORCE_AGREEMENT"
    
    # === 阶段4: VLM兜底 ===
    return "UNKNOWN"
```

### 9.7 发票在当前业务中的验证作用

从系统截图看，**商品房复审页面**有"增值税发票信息"区域：
- 增值税税额
- 不含增值税金额

#### 验证规则

```
发票验证（商品房）:
  不含税金额 + 税额 = 合同价格（或合同价税合计）
  购买方名称 = 合同买受人
  销售方名称 = 合同出卖人（开发商）
```

**存量房**通常不需要发票（二手房交易，卖方是个人，一般不开具增值税发票）
