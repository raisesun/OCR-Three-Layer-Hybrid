# 代码结构专项审查技术方案

## 审查目标

系统检查代码质量问题，包括重复代码、魔法数字、函数参数、复杂度、代码组织。

## 审查维度

### 1. 重复代码检测

**目标**：识别违反DRY原则的重复代码块

**检查方法**：
- **工具辅助**：使用`jdupes`或`fdclone`检测完全重复的文件
- **模式匹配**：识别相似的函数/代码块（相似度>70%）
- **逻辑重复**：识别相同逻辑的不同实现

**重点关注**：
- 各文档类型的提取逻辑是否有重复模式？
- VLM调用逻辑是否重复？
- 错误处理代码是否重复？
- 配置加载逻辑是否重复？

**苏格拉底提问**：
- 这段代码和那段代码为什么相似？
- 能否提取为公共函数？
- 提取后的接口应该如何设计？

### 2. 魔法数字提取

**目标**：识别硬编码的数字，建议常量化

**检查方法**：
- **正则扫描**：搜索所有数字字面量（排除0、1、-1等常用值）
- **上下文分析**：判断数字是否有业务含义
- **分类标记**：将魔法数字分类（超时、端口、阈值、尺寸等）

**重点关注**：
- 超时时间（如120.0秒、8192）
- 端口号（如8080、8082）
- 模型参数（如max_tokens=1024、temperature=0.1）
- 图像处理参数（如max_side=2000、quality=75）
- 置信度阈值（如0.6、0.8）

**苏格拉底提问**：
- 这个数字为什么是硬编码的？
- 它代表什么业务含义？
- 是否需要配置化？
- 不同环境是否需要不同的值？

### 3. 函数参数分析

**目标**：检查函数参数数量和合理性

**检查方法**：
- **参数数量统计**：统计每个函数的参数数量
- **参数类型检查**：检查是否有过多基本类型参数（应使用dataclass）
- **可选参数检查**：检查是否有过多可选参数（可能表示职责不清）

**标准**：
- ⚠️ 警告：参数数量>5
- ❌ 错误：参数数量>7
- ⚠️ 警告：超过3个可选参数

**苏格拉底提问**：
- 这个函数为什么需要这么多参数？
- 能否将相关参数封装为对象？
- 参数的默认值是否合理？
- 是否有参数可以从上下文中获取？

### 4. 复杂度分析

**目标**：识别过于复杂的函数

**检查方法**：
- **工具分析**：使用`radon`计算圈复杂度
- **行数统计**：统计函数行数
- **嵌套深度**：检查if/for/while嵌套层次

**标准**：
- ⚠️ 警告：圈复杂度>10
- ❌ 错误：圈复杂度>20
- ⚠️ 警告：函数行数>50
- ❌ 错误：函数行数>100
- ⚠️ 警告：嵌套深度>3

**苏格拉底提问**：
- 这个函数为什么这么复杂？
- 能否拆分为更小的函数？
- 复杂度来源是什么？（条件分支？循环？异常处理？）
- 拆分后的接口如何设计？

### 5. 代码组织评估

**目标**：评估模块内聚性和耦合度

**检查方法**：
- **模块职责分析**：每个模块是否职责单一？
- **依赖方向分析**：依赖是否单向？有无循环依赖？
- **接口分析**：模块间通过什么接口交互？

**关注点**：
- **内聚性**：模块内的函数是否相关？
- **耦合度**：模块间的依赖是否最小化？
- **稳定性**：修改一个模块是否会影响其他模块？

**苏格拉底提问**：
- 这个模块的职责是什么？
- 是否有不相关的功能混在一起？
- 模块间的依赖是否必要？
- 能否通过接口解耦？

## 实施计划

### 阶段1：工具准备（30分钟）

1. **安装分析工具**
   ```bash
   pip3 install radon lizard jdupes
   ```

2. **配置分析参数**
   - 确定检查范围（src/目录）
   - 确定排除规则（__pycache__、*.pyc）
   - 确定阈值标准

### 阶段2：重复代码检测（1小时）

1. **文件级重复检测**
   ```bash
   jdupes -r src/ > reports/duplicate_files.txt
   ```

2. **代码块相似度分析**
   - 使用lizard分析代码块
   - 手动检查相似函数
   - 记录重复代码位置和相似度

3. **生成报告**
   - 重复代码清单
   - 重复原因分析
   - 重构建议

### 阶段3：魔法数字提取（1小时）

1. **数字字面量扫描**
   ```bash
   grep -rn '[0-9]\+\.[0-9]\+' src/ > reports/magic_numbers.txt
   ```

2. **上下文分析**
   - 逐个检查数字的业务含义
   - 分类标记（配置、阈值、尺寸等）
   - 判断是否需要常量化

3. **生成报告**
   - 魔法数字清单
   - 建议的常量名
   - 配置化建议

### 阶段4：函数参数分析（1小时）

1. **参数数量统计**
   ```bash
   lizard src/ -CCN --csv > reports/function_params.csv
   ```

2. **参数合理性检查**
   - 识别参数过多的函数
   - 分析参数类型分布
   - 评估参数设计

3. **生成报告**
   - 参数过多的函数清单
   - 重构建议（封装为dataclass）
   - 接口优化建议

### 阶段5：复杂度分析（1小时）

1. **圈复杂度分析**
   ```bash
   radon cc src/ -s -a > reports/complexity.txt
   ```

2. **函数行数统计**
   ```bash
   lizard src/ -sL --csv > reports/function_lines.csv
   ```

3. **嵌套深度检查**
   - 使用正则检查嵌套层次
   - 识别深层嵌套的代码块

4. **生成报告**
   - 高复杂度函数清单
   - 拆分建议
   - 重构优先级

### 阶段6：代码组织评估（1小时）

1. **模块依赖分析**
   ```bash
   pydeps src/ --cluster --max-bullets=5 > reports/dependencies.png
   ```

2. **内聚性分析**
   - 检查每个模块的函数列表
   - 评估函数相关性
   - 识别不相关功能

3. **耦合度分析**
   - 检查模块间的import关系
   - 评估依赖必要性
   - 识别循环依赖

4. **生成报告**
   - 模块依赖图
   - 内聚性评估
   - 耦合度评估
   - 重构建议

### 阶段7：综合报告（1小时）

1. **汇总所有发现**
   - 重复代码统计
   - 魔法数字统计
   - 函数参数统计
   - 复杂度统计
   - 代码组织评估

2. **优先级排序**
   - 按影响程度排序
   - 按修复难度排序
   - 按紧急程度排序

3. **生成改进计划**
   - 短期修复（1周内）
   - 中期重构（1个月内）
   - 长期优化（3-6个月）

## 预期输出

1. **代码结构审查报告**（reports/code_structure_review.md）
   - 重复代码清单
   - 魔法数字清单
   - 函数参数分析
   - 复杂度分析
   - 代码组织评估

2. **改进建议清单**（reports/improvement_suggestions.md）

   - 预期收益和成本

   - 复杂度热力图（PNG）

- **风险**：发现大量问题需要重构
- **约束**：不能影响生产环境稳定性
- **缓解**：渐进式改进，优先修复高风险问题

- 提取出至少20个魔法数字
- 分析出至少5个参数过多的函数
- 识别出至少5个高复杂度函数
- 生成完整的代码质量报告
- 提出可执行的改进计划
