## 问题2：PaddleOCR-VL-0.9B vs GLM-OCR 对比

### 2.1 模型规格对比

| 项目 | GLM-OCR (当前) | PaddleOCR-VL-0.9B |
|------|---------------|-------------------|
| 参数量 | ~3B（估计） | 0.9B |
| 模型大小 | 950 MB (Q8_0 GGUF) + 484 MB mmproj = **1.4 GB** | ~1.8 GB (safetensors) |
| 基础语言模型 | 未知 | ERNIE-4.5-0.3B |
| 视觉编码器 | 独立mmproj | NaViT动态分辨率 |
| 量化 | Q8_0 (GGUF) | FP16/BF16 |
| 支持语言 | 中英 | **109种语言** |
| 训练数据 | 证件OCR | 文档解析（论文/报表/证书） |

### 2.2 预期效果对比

| 能力 | GLM-OCR | PaddleOCR-VL-0.9B | 说明 |
|------|---------|-------------------|------|
| 基础文本识别 | 中等 | **更好** | NaViT动态分辨率，更适合文档 |
| 表格理解 | 弱 | **更强** | 专门训练了表格识别 |
| 证件理解 | 中等 | ⚠️ 未知 | 主要训练在文档页面，非证件 |
| 列错位修复 | ❌ 不行 | ⚠️ 可能 | VLM理论上能理解列关系 |
| 数字补全 | ❌ 不行 | ⚠️ 可能 | VLM能理解身份证号18位约束 |
| 形近字纠正 | ❌ 不行 | ⚠️ 可能 | VLM能理解上下文语义 |

### 2.3 关键差异分析

**PaddleOCR-VL-0.9B 的优势：**
1. **文档解析SOTA** — 在公开benchmark上达到SOTA，远超同类轻量模型
2. **109种语言** — 多语言支持更广
3. **NaViT动态分辨率** — 不损失细节地处理不同尺寸文档
4. **表格/公式/图表** — 专门训练了复杂文档元素识别

**PaddleOCR-VL-0.9B 的劣势：**
1. **训练数据偏向文档页面** — 论文、报表、合同等，不是证件
2. **模型不完整** — 当前 PaddleOCR 3.7.0 pipeline无法加载（缺少inference权重）
3. **需要自行集成** — 不能直接用 PaddleOCR pipeline，需要用 transformers/mindnlp 加载

**GLM-OCR 的特点：**
1. **已经集成** — 通过 llama-server 运行，已有完整的调用链
2. **针对证件优化** — 系统提示词专门设计为"证件OCR助手"
3. **但能力不足** — 4个错误全部无法修复

### 2.4 建议

| 方案 | 投入 | 预期效果 | 推荐度 |
|------|------|---------|--------|
| 继续用 GLM-OCR | 0 | 无法修复错误 | ❌ |
| 修复 PaddleOCR-VL-0.9B 集成 | 1-2天 | 可能修复1-2个错误 | ⚠️ |
| **直接用 Qwen2.5-VL-7B** | 1天 | 大概率修复3-4个错误 | ✅ 推荐 |
| doubao-seed-2.0-pro (云端) | 0.5天 | 大概率修复3-4个错误 | ✅ 但需处理PII |

**结论：PaddleOCR-VL-0.9B 理论上比 GLM-OCR 更强，但集成成本高且效果不确定。建议直接上 Qwen2.5-VL-7B。**

## 问题3：AI-OCR 位置感知提取方案

### 3.1 AI-OCR 项目思路

AI-OCR 项目（`/Users/dongsun/Github/AI-OCR/`）使用了一种完全不同的方法：**基于坐标位置的字段提取**。

核心思路：
```
1. 整页 OCR → 得到每个文本行的 (text, bbox, score)
2. 定义字段搜索区域（相对坐标 0-1）
   例如：户主姓名区域 = (0.0, 0.1, 0.5, 0.2) 表示左上角20%高度的区域
3. 在区域内找锚点关键词（如"户主姓名"）
4. 根据锚点位置，按偏移量定位数据单元格
   例如：数据在锚点右侧 offset_x=(0.15, 0.35), 同行 tolerance_y=0.02
5. 提取单元格文本
```

关键配置（AI-OCR AWB 提取器）：
```python
# 搜索区域（相对坐标 x1, y1, x2, y2）
HEADER_SEARCH_REGIONS = {
    'pieces':         (0.03, 0.35, 0.25, 0.60),
    'gross_weight':   (0.08, 0.35, 0.35, 0.60),
    'chargeable_weight': (0.20, 0.35, 0.55, 0.60),
}

# 数据行偏移
CELL_ROW_OFFSET = (0.02, 0.25)  # 数据在表头下方 2%-25% 处

# 列容差
CELL_COL_TOLERANCE = 0.08  # 同一列的X坐标差异容忍度
```

### 3.2 对户口本的适用性分析

**户口本首页布局（相对坐标估算）：**
```
┌─────────────────────────────────┐
│  注意事项 (0.0, 0.0, 1.0, 0.3)  │
│  ...                            │
├────────┬────────┬───────────────┤
│ 户别   │ 非农业 │ (0.0, 0.35)   │
├────────┼────────┼───────────────┤
│ 户主   │ 姓名   │ 王晨露 ← 第2列 │
│ 姓名   │        │ (0.5, 0.40)   │
├────────┼────────┼───────────────┤
│ 户号   │ 005300251              │
│ (0.0, 0.50)                     │
├────────┼───────────────────────-┤
│ 住址   │ 安徽省蚌埠市...         │
│ (0.0, 0.60)                     │
└────────┴────────────────────────┘
```

**列错位问题的根因**：
- "户主姓名" 这个标签在左列 (x≈0.0)
- "王晨露" 在右列 (x≈0.5)
- 但 GLM-OCR 把两列数据合并成一行输出 → 丢失了坐标信息

**如果用位置感知提取**：
```python
# 1. OCR获取所有文本行 + 坐标
ocr_items = ocr_with_coords(image)

# 2. 找到"户主姓名"标签的位置
label = find_item_in_region(ocr_items, region=(0.0, 0.35, 0.3, 0.50), text_pattern="户主姓名")

# 3. 在标签右侧区域找数据
data = find_item_in_region(ocr_items, region=(0.4, 0.35, 0.8, 0.50))
# data.text = "王晨露" ← 不会混入左列的"王子龙"！
```

### 3.3 可行性评估

| 方面 | 评估 | 说明 |
|------|------|------|
| **技术可行性** | ✅ 完全可行 | PaddleOCR 已输出坐标 (rec_polys, rec_boxes) |
| **修复列错位** | ✅ 大概率可行 | 基于坐标区域提取，不依赖行合并 |
| **修复数字漏识别** | ❌ 不行 | 坐标方法不能修复OCR字符识别错误 |
| **修复形近字混淆** | ❌ 不行 | 同上 |
| **开发成本** | 中（2-3天） | 需要为每种证件类型定义坐标区域 |
| **维护成本** | 中 | 不同扫描件的布局可能有偏差 |

### 3.4 关键问题：PaddleOCR 已输出坐标！

**重要发现**：PaddleOCR PP-OCRv6 的输出**已包含坐标信息**：
```python
result = ocr.predict(input=image_path)
r = result[0]
r['rec_texts']    # 文本内容
r['rec_polys']    # 多边形坐标（4个顶点）
r['rec_boxes']    # 矩形边界框 [x1, y1, x2, y2]
r['rec_scores']   # 置信度
```

这意味着我们可以**不改变OCR引擎**，只改变提取策略：
1. 继续用 PP-OCRv6 做文本识别（速度2-3秒）
2. 用坐标位置做字段定位（而不是用全文正则匹配）
3. 对数字字段仍用正则，但对位置敏感字段用坐标区域

### 3.5 位置感知提取方案设计

```
                    ┌─────────────────┐
                    │  PP-OCRv6 OCR   │
                    │  (文本+坐标)     │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
    ┌─────────▼──────┐ ┌────▼─────┐ ┌──────▼───────┐
    │ 位置感知提取    │ │ 正则提取  │ │ VLM兜底      │
    │ (坐标区域匹配) │ │ (全文)   │ │ (异常触发)   │
    │                │ │          │ │              │
    │ 适用：         │ │ 适用：   │ │ 适用：       │
    │ - 户主姓名     │ │ - 身份证号│ │ - 校验失败   │
    │ - 住址        │ │ - 日期   │ │   的字段     │
    │ - 合同编号     │ │ - 金额   │ │              │
    └────────┬───────┘ └────┬─────┘ └──────┬───────┘
             │              │              │
             └──────────────┼──────────────┘
                            │
                  ┌─────────▼─────────┐
                  │  后处理校验        │
                  │  - 身份证18位      │
                  │  - 省份白名单      │
                  │  - 日期格式        │
                  └───────────────────┘
```
