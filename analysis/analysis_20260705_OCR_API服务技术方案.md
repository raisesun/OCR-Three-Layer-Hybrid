# OCR API 服务技术方案分析

## 分析日期: 2026-07-05

## 一、第一性原理思考

### 1.1 问题的本质是什么？

**核心问题**：将现有的OCR本地处理能力转化为可被外部系统调用的服务。

**不可简化的事实**：
1. 输入：图片/PDF文件（二进制数据）
2. 处理：OCR文本提取 → 文档分类 → 字段提取
3. 输出：结构化JSON结果
4. 约束：需要鉴权、需要稳定、需要应对并发

### 1.2 剥离假设和惯例

**常见假设**：
- 假设1：需要微服务架构 → 质疑：初期是否真的需要？
- 假设2：需要Kubernetes → 质疑：流量规模是否达到？
- 假设3：需要OAuth2 → 质疑：内部系统调用是否需要这么重？
- 假设4：需要消息队列 → 质疑：同步处理是否够用？

**从物理本质出发**：
- 本质是一个 HTTP 服务器，接收文件，返回JSON
- 本质需要：路由 + 鉴权 + 限流 + 日志 + 监控
- 本质挑战：文件上传的内存管理、长时间处理的超时处理

## 二、批判性思维

### 2.1 质疑前提假设

| 假设 | 质疑 | 结论 |
|------|------|------|
| "需要支持高并发" | 实际并发量是多少？内部系统通常<10 QPS | 分阶段设计，初期单机即可 |
| "需要公网部署" | 是否真的需要？还是先局域网？ | 先局域网，预留公网能力 |
| "需要PDF支持" | PDF是图片还是文字？处理策略不同 | 明确PDF类型，分别处理 |
| "需要实时返回" | 大批量是否需要异步？ | 同步+异步双模式 |

### 2.2 区分事实与观点

**事实**：
- 现有代码是Python，使用PP-OCRv6 + VLM
- 单张图片处理时间：规则层3-6秒，VLM层30-120秒
- 当前是本地运行，无并发处理

**观点**：
- "需要用FastAPI" → 可选，Flask/Starlette也可以
- "需要用Redis" → 可选，取决于是否需要分布式
- "需要用Docker" → 推荐，但不是必须

### 2.3 识别逻辑谬误

**谬误1**："因为要支持高并发，所以需要微服务"
- 反驳：高并发可以通过负载均衡+多实例解决，不一定需要微服务拆分

**谬误2**："因为要公网部署，所以需要复杂的鉴权"
- 反驳：API Key + HTTPS 对大多数场景足够

**谬误3**："因为处理时间长，所以需要消息队列"
- 反驳：同步API + 异步API双模式可以覆盖大部分场景

## 三、苏格拉底提问法（自问自答）

### 3.1 服务器搭建

**Q: 为什么选择FastAPI而不是Flask？**
A: 
- FastAPI原生支持异步，适合I/O密集型任务
- 自动API文档生成（Swagger/ReDoc）
- 类型提示支持，减少错误
- 性能优于Flask（基于Starlette）

**Q: 为什么需要Docker？**
A:
- 环境一致性：开发、测试、生产环境相同
- 依赖隔离：PaddleOCR、VLM服务依赖复杂
- 部署简化：一键启动，无需手动安装依赖
- 扩展方便：Kubernetes/Docker Swarm原生支持

**Q: 为什么需要反向代理（Nginx）？**
A:
- SSL终止：HTTPS在Nginx层处理
- 负载均衡：多实例时分发请求
- 静态文件：提供API文档等静态资源
- 请求限流：第一层防护

### 3.2 鉴权方案

**Q: 为什么选择API Key而不是OAuth2？**
A:
- 内部系统调用：API Key足够简单
- 实施成本低：无需OAuth2服务器
- 管理简单：数据库/配置文件存储
- 后续可扩展：可升级到OAuth2

**Q: API Key如何安全管理？**
A:
- 存储：数据库存储hash值，不存明文
- 传输：仅通过HTTPS
- 权限：支持权限分级（只读/读写）
- 审计：记录每次使用的API Key

**Q: 是否需要用户认证？**
A:
- 初期：不需要，API Key标识调用方
- 后期：如需用户级权限，可叠加JWT

### 3.3 健壮性与稳定性

**Q: 如何处理长时间处理（VLM层30-120秒）？**
A:
- 方案1：同步API，设置长超时（180秒）
- 方案2：异步API，返回任务ID，查询结果
- 推荐：双模式，小批量同步，大批量异步

**Q: 如何处理服务崩溃？**
A:
- 进程级：Supervisor/systemd自动重启
- 容器级：Docker restart policy
- 应用级：健康检查端点 `/health`
- 监控级：Prometheus + Grafana告警

**Q: 如何处理VLM服务不可用？**
A:
- 降级：自动切换到规则层
- 熔断：连续失败N次后，暂时禁用VLM
- 告警：通知运维人员

**Q: 如何防止内存泄漏？**
A:
- 文件上传：使用临时文件，处理完删除
- 模型加载：全局单例，不重复加载
- 监控：定期检查内存使用

### 3.4 高并发与高流量

**Q: 实际并发量会是多少？**
A:
- 内部系统：通常<10 QPS
- 公测阶段：可能10-100 QPS
- 峰值：可能短暂100-500 QPS

**Q: 单机能否支撑？**
A:
- 规则层：3-6秒/张，单机约0.2-0.3 QPS
- VLM层：30-120秒/张，单机约0.01-0.03 QPS
- 结论：单机不支持高并发，需要多实例

**Q: 如何水平扩展？**
A:
- 无状态设计：API层不存储状态
- 共享存储：文件存储使用共享磁盘/NFS
- 负载均衡：Nginx分发到多实例
- 会话亲和：不需要，请求独立

**Q: 是否需要消息队列？**
A:
- 异步任务：使用数据库任务表（简单）或Redis Queue
- 优先级：普通/加急队列
- 限流：队列长度限制，超过返回错误

**Q: 如何限流保护？**
A:
- 第一层：Nginx限流（按IP）
- 第二层：应用限流（按API Key）
- 第三层：队列限流（总任务数）

## 四、方案对比

### 4.1 服务器架构方案

| 方案 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **方案A: 单机FastAPI** | 单台服务器运行FastAPI | 简单、快速上线 | 单点故障、扩展受限 | 内部使用、<10 QPS |
| **方案B: Docker多实例** | Docker Compose多实例+Nginx | 易扩展、环境一致 | 需要Docker知识 | 中等流量、<100 QPS |
| **方案C: Kubernetes** | K8s集群自动扩缩容 | 高可用、自动扩展 | 复杂、运维成本高 | 大规模、>100 QPS |

**推荐**：阶段式演进
- 初期：方案A（快速验证）
- 成长期：方案B（平衡复杂度）
- 成熟期：方案C（按需扩展）

### 4.2 鉴权方案

| 方案 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **方案A: API Key** | Header中传递API Key | 简单、易实施 | 安全性一般 | 内部系统、可信网络 |
| **方案B: JWT** | JSON Web Token | 无状态、可传递用户信息 | 需要认证服务器 | 需要用户认证 |
| **方案C: OAuth2** | 标准授权框架 | 安全性高、生态成熟 | 实施复杂 | 第三方接入 |

**推荐**：方案A（API Key）+ HTTPS
- 内部系统调用足够
- 后续可扩展JWT/OAuth2

### 4.3 异步处理方案

| 方案 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **方案A: 同步API** | 请求-响应模式 | 简单、实时 | 长时间阻塞 | 单张/少量图片 |
| **方案B: 任务队列** | 提交任务→查询结果 | 支持大批量、解耦 | 需要轮询/回调 | 大批量处理 |
| **方案C: WebSocket** | 长连接实时推送 | 实时性好 | 实现复杂 | 需要实时反馈 |

**推荐**：方案A + 方案B双模式
- 同步API：1-10张图片
- 异步API：>10张图片

### 5.2 技术栈

| 组件 | 技术选择 | 理由 |
|------|----------|------|
| Web框架 | FastAPI | 异步支持、自动文档、类型安全 |
| 反向代理 | Nginx | SSL、限流、负载均衡 |
| 容器化 | Docker | 环境一致、易部署 |
| 数据库 | SQLite/PostgreSQL | 任务存储、API Key管理 |
| 缓存 | Redis (可选) | 限流计数、会话存储 |
| 监控 | Prometheus + Grafana | 指标采集、可视化 |
| 日志 | Python logging + ELK | 结构化日志、集中分析 |

### 5.5 鉴权设计

#### API Key 管理
```python
# 数据库表结构
class APIKey(Base):
    id: int
    key_hash: str      # bcrypt hash
    key_prefix: str    # 前4位，用于显示
    name: str          # 应用名称
    permissions: list  # ["ocr.read", "ocr.write"]
    rate_limit: int    # 每分钟请求数
    is_active: bool
    created_at: datetime
    last_used_at: datetime
```

#### 验证流程
```
1. 从 Header 提取 API Key: Authorization: Bearer {key}
2. 查询数据库获取 key_hash
3. bcrypt 验证 key 与 key_hash
4. 检查 is_active 状态
5. 检查 rate_limit (Redis计数)
6. 记录 last_used_at
```

#### 健康检查

GET /health

Response:

  "checks": {
    "database": "ok",

#### 熔断降级
```python
# VLM服务熔断器
class VLMCircuitBreaker:
    failure_threshold: int = 5      # 连续失败5次
    recovery_timeout: int = 60      # 60秒后尝试恢复
    state: str = "closed"           # closed/open/half-open
    
    # 当VLM不可用时，自动降级到规则层
```

#### 限流策略
```
第一层: Nginx 按IP限流
  - 10 requests/second
  
第二层: 应用层按API Key限流
  - 根据API Key配置的rate_limit
  
第三层: 任务队列限流
  - 最大队列长度: 1000
  - 超过返回 429 Too Many Requests
```

## 八、总结

### 核心设计原则
1. **简单优先**: 初期不过度设计，按需演进
2. **无状态设计**: API层无状态，便于水平扩展
3. **双模式支持**: 同步+异步覆盖不同场景
4. **降级优先**: 依赖服务不可用时自动降级
5. **可观测性**: 完善的日志、监控、告警

### 关键技术决策
1. **FastAPI**: 异步支持、自动文档
2. **API Key**: 简单有效的鉴权
3. **Docker**: 环境一致、易部署
4. **双模式API**: 同步+异步
5. **熔断降级**: 保证核心功能可用
