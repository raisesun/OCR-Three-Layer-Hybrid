## 二、方案详情

### 方案 A：PP-ChatOCRv4 本地化 ⭐ 推荐

**项目位置**：`/Users/dongsun/Github/OCR-PP-CharOCRv4/`
**参考文档**：`PP-ChatOCRv4本地化部署详细指南.md`

#### 核心思路

PP-ChatOCRv4 默认使用百度千帆 API（ERNIE + Embedding）。源码分析发现其架构天然支持 OpenAI 兼容接口——只需修改 YAML 配置文件，将 `base_url` 指向本地 Ollama 即可实现完全离线。

#### 技术栈

| 组件 | 默认 | 本地化 |
|------|------|--------|
| OCR | PP-OCRv4 (本地) | PP-OCRv4 (本地) ✅ |
| 版面分析 | RT-DETR (本地) | RT-DETR (本地) ✅ |
| 表格识别 | SLANet+ (本地) | SLANet+ (本地) ✅ |
| 印章识别 | 专用模型 (本地) | 专用模型 (本地) ✅ |
| Embedding | 百度千帆 embedding-v1 (云端) | Ollama nomic-embed-text (本地) |
| LLM | ERNIE-3.5 (云端) | Ollama qwen2.5:1.5b (本地) |
| 向量检索 | FAISS (本地) | FAISS (本地) ✅ |

#### 优势

1. **开发成本最低**：仅改 6 行 YAML 配置
2. **功能最完备**：内置版面分析、表格识别、印章识别、向量检索、Prompt 工程
3. **无需写正则**：只需指定 `key_list`，Prompt 由百度团队优化
4. **向量检索增强**：长文档也能精准定位关键信息

#### 风险

1. **未经实测**：需要真实样本验证准确率
2. **Qwen2.5-1.5B 能力未知**：对复杂 Prompt 的理解可能不如 ERNIE-3.5
3. **nomic-embed-text 中文效果**：可能不如 bge-large-zh

#### 预估性能

| 文档类型 | 预计耗时 |
|----------|:--------:|
| 身份证（短文本） | 3-8s |
| 发票（表格） | 5-15s |
| 购房合同（长篇） | 15-60s |

### 方案 B：GLM-OCR ❌ CPU 不可用

**项目位置**：`/Users/dongsun/Github/OCR-glm-Mac/`
**参考文档**：`GLM-OCR-技术文档.md`, `实测结果与方案总结.md`

#### 核心思路

使用智谱 Z.ai 开源的 GLM-OCR 0.9B 多模态模型，端到端直接从图片提取结构化 JSON。

#### 技术栈

| 组件 | 模型 | 大小 |
|------|------|:----:|
| 视觉编码器 | CogViT (24 层) | - |
| 语言解码器 | GLM-0.5B (16 层) | - |
| 整体 | GLM-OCR 0.9B | ~2.65 GB (BF16) |
| 版面分析 | PP-DocLayoutV3 | ~200 MB |

#### 实测结果（关键数据）

| 配置 | 单张耗时 | 评价 |
|------|:--------:|:----:|
| CPU + float32 | **338 秒** | ❌ 不可接受 |
| CPU + bfloat16 | **>30 分钟** | ❌ 卡死 |
| MPS (Apple GPU) + bfloat16 | **内存不足** | ❌ 17GB+ 缓冲区需求 |

**准确率测试**：
- 身份证背面：✅ 完全正确（签发机关、有效期限）
- 购房合同签字页：⚠️ 识别出公司名称（Prompt 不匹配场景）

#### 结论

**准确率优秀，但 CPU 推理速度完全不可接受**（5-6 分钟/张）。在 Apple M4 上 MPS 也失败（内存不足）。

**推荐出路**：
- 云 API（Z.ai，$0.03/百万 tokens）
- 等 Ollama 量化版本（30-60 秒/张）
- 需要 NVIDIA GPU（1-3 秒/张）

### 方案 C：PaddleOCR 纯规则 ✅ 已验证可用

**项目位置**：`/Users/dongsun/Github/OCR-Paddle-Mac/`
**参考文档**：`README.md`, `docs-angus/中文证件票据识别-技术方案.md`

#### 核心思路

纯 PaddleOCR（PP-OCRv6）做 OCR，再用正则表达式做结构化提取。

#### 技术栈

| 组件 | 模型 | 大小 |
|------|------|:----:|
| 文本检测 | PP-OCRv6_medium_det | < 100 MB |
| 文本识别 | PP-OCRv6_medium_rec | < 100 MB |
| 提取层 | 正则表达式 + 关键词匹配 | 0 MB |

#### 实测结果（关键数据）

| 指标 | 数值 | 评价 |
|------|:----:|:----:|
| 单张 OCR 耗时 | **3-8 秒** | ✅ 可接受 |
| 字段提取耗时 | **< 1 毫秒** | ✅ 极快 |
| 身份证准确率 | **100%** (6/6 字段) | ✅ 完美 |
| 结婚证准确率 | **100%** (8/8 字段) | ✅ 完美 |
| 发票准确率 | 待验证 | ⏳ |
| 合同类准确率 | **低**（格式不固定） | ❌ |

#### 优势

1. **速度最快**：3-8 秒/张，生产环境可用
2. **准确率最高**（固定格式）：身份证、结婚证 100%
3. **完全离线**：无任何外部依赖
4. **资源占用最小**：无 LLM、无 VLM

#### 风险

1. **无法处理非固定格式**：购房合同、租房合同等长篇文档无能为力
2. **正则规则维护成本**：每种文档都要写一套正则，且需针对不同版本调整
3. **对 OCR 错误敏感**：一旦 OCR 识别错字，正则就匹配失败

#### 适用文档

| 文档类型 | 是否适用 | 原因 |
|----------|:--------:|------|
| 身份证 | ✅ | 格式全国统一 |
| 结婚证 | ✅ | 格式全国统一 |
| 离婚证 | ✅ | 格式全国统一 |
| 发票 | ✅ | 增值税发票格式标准化 |
| 快递单 | ✅ | 运单号/电话/地址模式固定 |
| 户口本 | ⚠️ | 格式较固定但各省略有差异 |
| 房产证 | ❌ | 各省格式不统一 |
| 购房合同 | ❌ | 长篇 + 格式不统一 |
| 租房合同 | ❌ | 格式不统一 |
| 提单 | ❌ | 航运公司格式各异 |

### 方案 D：混合架构 ⏳ 部分完成

**项目位置**：`/Users/dongsun/Github/OCR-PaddleOCR-VL-0.9b/`
**参考文档**：`混合架构方案-详细说明.md`

#### 核心思路

三层分流：
- **3A 层（规则）**：身份证、结婚证、发票、快递单 → 正则
- **3B 层（VLM）**：户口本、房产证、提单 → PaddleOCR-VL-0.9B
- **3C 层（LLM）**：购房合同、租房合同、协议书 → Qwen2.5:1.5B

#### 技术栈

| 层 | 工具 | 大小 | 用途 |
|---|------|:----:|------|
| OCR 层 | PaddleOCR PP-OCRv6 | ~200 MB | 文字识别 + 版面分析 |
| 3A 规则层 | 正则表达式 | 0 MB | 固定格式文档 |
| 3B VLM 层 | PaddleOCR-VL-0.9B | ~1.94 GB | 半固定格式文档 |
| 3C LLM 层 | Qwen2.5:1.5B (Ollama) | ~1 GB | 长篇非固定文档 |

#### 实测结果

| 组件 | 状态 | 说明 |
|------|:----:|------|
| OCR 层 (PaddleOCR) | ✅ 完成 | 3-8 秒/张 |
| 3A 规则层 | ✅ 完成 | 身份证 100%、结婚证 100% |
| 自动文档分类 | ✅ 完成 | 关键词匹配 |
| 3B VLM 层 | ❌ 阻塞 | SSL 证书问题导致模型下载失败 |
| 3C LLM 层 | ⏳ 待实施 | 代码已写，待安装 Ollama |

#### 优势

1. **每种文档用最合适的工具**：不浪费算力
2. **规则保底**：即使 LLM/VLM 故障，固定证件仍可处理
3. **渐进式部署**：先上规则，再加 VLM，最后加 LLM
4. **成本最优**：CPU 即可运行

#### 风险

1. **VLM 层阻塞**：PaddleOCR-VL-0.9B 模型下载遇到 SSL 问题
2. **工程复杂度高**：三层架构需要维护多套代码
3. **VLM CPU 性能**：PaddleOCR-VL-0.9B (1.94GB) 在 CPU 上可能较慢

## 四、关键决策点

### 4.1 为什么 GLM-OCR 被排除？

**核心数据**：CPU 推理 338 秒/张，完全不可接受。

即使准确率很高（身份证 100%），但 5-6 分钟的处理时间在商业场景下毫无价值。MPS（Apple GPU）也因内存不足失败（需要 17GB+ 缓冲区）。

**结论**：除非部署到 NVIDIA GPU 服务器，否则 GLM-OCR 不适合当前约束。

### 4.2 为什么纯规则方案不够？

**覆盖范围有限**：只能处理固定格式文档（身份证、结婚证、发票、快递单），对非固定格式文档（购房合同、房产证、提单）无能为力。

但**作为第一阶段快速上线方案是最佳选择**——速度快、准确率高、资源占用低。

### 4.3 为什么混合架构没有 PP-ChatOCRv4 本地化方案好？

**两者本质上做的是同一件事**：OCR → 提取。但实现方式不同：

| 维度 | 混合架构 (D) | PP-ChatOCRv4 本地化 (A) |
|------|:------------:|:-----------------------:|
| OCR 层 | PaddleOCR PP-OCRv6 | PaddleOCR PP-OCRv4 (相同) |
| 提取层 | 自己写三层逻辑 | 复用官方 Prompt 工程 |
| 向量检索 | ❌ 无 | ✅ FAISS 内置 |
| 表格处理 | 需自己写逻辑 | ✅ 内置专用 Prompt |
| 印章处理 | 需自己写逻辑 | ✅ 内置 OCR |
| 文档分类 | 需自己写 | 可直接指定 doc_type |
| 维护成本 | 高（三层代码） | 低（配置驱动） |

**关键差异**：
- **方案 D 需要自己造轮子**：写规则、写 VLM 调用逻辑、写 LLM 调用逻辑、写分类器
- **方案 A 直接复用百度团队的成果**：Prompt 工程、表格处理、印章识别都已优化

### 4.4 PP-ChatOCRv4 本地化的最大风险是什么？

**Qwen2.5-1.5B 的理解能力**。

百度原版用 ERNIE-3.5-8k（80 亿参数），我们本地化用 Qwen2.5-1.5b（15 亿参数）。参数量差距 5 倍以上。

**缓解措施**：
1. 先用 1.5B 做原型验证
2. 如果准确率不够，升级到 3B 或 7B
3. 必要时混合使用：规则处理固定文档，LLM 处理长文档

## 五、推荐路线

### 5.1 短期（1-2 周）：快速上线 + 验证

**采用方案 A（PP-ChatOCRv4 本地化）+ 方案 C（纯规则）兜底**

```
第一步（1-2 天）：搭建 PP-ChatOCRv4 本地化原型
├─ 安装 Ollama + PaddleX
├─ 修改 YAML 配置
└─ 用真实样本验证准确率

第二步（1-2 天）：对比测试
├─ 同时跑方案 C（纯规则）
├─ 对比两者准确率
└─ 决定哪些文档用规则，哪些用 LLM

第三步（3-5 天）：针对性优化
├─ 准确率不达标的文档类型单独优化
├─ 必要时升级 LLM 模型
└─ 编写规则兜底逻辑
```

### 5.2 中期（2-4 周）：生产化

**基于实测结果，选择最优组合**

```
固定格式文档（身份证、结婚证、发票、快递单）
└─ 用方案 C 的纯规则（已验证 100% 准确率）
└─ 速度最快（< 1 秒）

半固定格式（户口本、房产证、提单）
└─ 用方案 A 的 PP-ChatOCRv4
└─ 无需维护正则规则

非固定格式（合同、协议书）
└─ 用方案 A 的 PP-ChatOCRv4
└─ 向量检索 + Prompt 工程
```

### 5.3 长期（1-3 个月）：持续优化

```
1. 收集真实业务样本，建立测试集
2. 针对准确率不达标的文档类型优化
3. 考虑 LLM 模型升级（3B → 7B）
4. 考虑 GPU 部署（如果业务量需要）
5. 考虑模型微调（针对特定业务场景）
```

5. ✅ 维护成本最低（配置驱动，无需写正则）

**方案 C（PaddleOCR 纯规则）作为兜底**：

- 对于方案 A 准确率不达标的固定格式文档，用方案 C 的正则逻辑兜底
- 方案 C 已经在 Mac 上验证可用，准确率高

### 7.3 明确放弃

**方案 B（GLM-OCR）在当前约束下不可行**：

- 仅当部署到 NVIDIA GPU 时再考虑
