## 一、问题发现

- 存量房业务：BBJZ-2025-1013085（15张图片）

**关键发现**：

## 二、LLM层职责分析

### 2.1 PP-ChatOCRv4 的职责

**核心定位**：RAG架构的文档问答系统

**工作流程**：
```
输入：文档图片
  ↓
[步骤1] 视觉信息提取 (visual_predict)
  ├─ OCR 文本识别
  ├─ 文档方向分类
  ├─ 文档扭曲矫正
  ├─ 表格识别
  ├─ 印章识别
  └─ 输出：visual_info（结构化视觉信息）
  ↓
[步骤2] 向量检索构建
  ├─ 将 OCR 文本分块
  ├─ 使用 nomic-embed-text 生成向量
  └─ 构建向量索引
  ↓
[步骤3] LLM 问答 (chat)
  ├─ 对于每个目标字段：
  │   ├─ 向量检索：找到与字段语义相关的文本块
  │   ├─ 构建 Prompt：相关文本块 + 目标字段
  │   └─ 调用 Qwen35-4B 提取字段
  └─ 输出：提取的字段
```

**设计初衷**：
- 适用于**开放式问答**场景
- 用户问"这份文档中关于违约责任的内容是什么？"
- 不需要预知答案位置，通过语义检索找到相关内容

### 2.2 Qwen35-4B 的职责

**核心定位**：语义理解 + 字段提取

**具体任务**：
- 接收 PP-ChatOCRv4 构建的 Prompt
- 理解文本块内容
- 提取目标字段的值
- 返回结构化结果

## 三、方案错配分析

### 3.1 合同提取的真实需求

**需求特征**：
1. **固定字段**：买受人、总价款、建筑面积等（已知要提取什么）
2. **多页文档**：字段分散在 5-50+ 页
3. **结构化格式**：合同有标准格式，字段位置相对固定
4. **批量处理**：需要从大量合同中提取相同的字段

### 3.2 PP-ChatOCRv4 与合同提取的错配

| 维度 | PP-ChatOCRv4 设计 | 合同提取需求 | 匹配度 |
|------|------------------|------------|--------|
| **任务类型** | 开放式问答 | 固定字段提取 | ❌ 错配 |
| **字段位置** | 未知，需要检索 | 已知，相对固定 | ❌ 错配 |
| **文档结构** | 单文档查询 | 多页文档 | ❌ 错配 |
| **处理速度** | 向量检索+LLM（慢） | 需要快速提取 | ❌ 错配 |
| **字段数量** | 1个问题→1个答案 | N个字段→N个答案 | ❌ 错配 |

**问题本质**：
- PP-ChatOCRv4 是**间接提取**：图片 → OCR → 文本分块 → 向量检索 → LLM提取
- 合同提取需要**直接提取**：图片 + 字段列表 → 直接识别提取
- 向量检索对于固定字段是多余的步骤

### 3.3 为什么 VLM 更适合

**VLM 直接提取**：
```
输入：合同页面图片 + 固定字段列表
  ↓
VLM 直接识别：
- "买受人"在图片哪里 → 提取值
- "总价款"在图片哪里 → 提取值
- "建筑面积"在图片哪里 → 提取值
  ↓
优势：直接、快速、不需要中间检索步骤
```

**对比**：
- **PP-ChatOCRv4**：图片 → OCR → 文本分块 → 向量检索 → LLM 提取（间接，多步骤，197-316秒）
- **VLM**：图片 + 字段列表 → 直接提取（直接，单步骤，50-100秒）

## 四、PP-ChatOCRv4 的正确定位

### 4.1 适合的场景

**1. 复杂表格的开放式问答**
```
场景：从财务报表中提取"2024年Q3的营业收入同比增长率"
特点：
- 字段不固定（用户问什么就提取什么）
- 需要理解表格结构和上下文
- 需要跨多个表格汇总信息
优势：向量检索可以找到相关内容，LLM 可以理解上下文
```

**2. 非结构化文档的信息检索**
```
场景：从100页的法律文档中找到"关于知识产权的条款"
特点：
- 文档长，信息分散
- 不知道具体内容在哪里
- 需要语义检索
优势：向量检索可以快速定位相关内容
```

**3. 需要推理的复杂提取**
```
场景：从合同中提取"违约责任的赔偿上限"
特点：
- 答案不在单一位置
- 需要理解多个条款的关系
- 需要推理计算
优势：LLM 可以进行推理和综合
```

### 4.2 不适合的场景

**1. 固定字段的批量提取**
```
场景：从100份合同中提取"买受人、总价款、建筑面积"
问题：
- 字段固定，不需要语义检索
- 每份合同都要做向量检索（浪费）
- 速度慢（197-316秒/文档）
更好方案：VLM 直接提取（50-100秒/文档）
```

**2. 多页文档的结构化提取**
```
场景：从50页的购房合同中提取7个固定字段
问题：
- PP-ChatOCRv4 是单文档查询，不支持多页
- 字段分散在多页，向量检索效果差
更好方案：逐页 VLM 提取 + 字段合并
```

**3. 需要高速处理的场景**
```
场景：生产环境需要快速处理大量合同
问题：
- PP-ChatOCRv4 太慢（197-316秒）
- 向量检索 + LLM 问答的开销太大
更好方案：VLM 直接提取（快 2-3 倍）
```

## 五、架构调整方案

### 5.1 当前架构

```
节点1: OCR文本提取
  ├─ PP-OCRv6（主力）
  └─ PaddleOCR-VL（备用）
  
节点2: 文档分类
  ├─ 规则分类器（主力）
  └─ VLM分类兜底（Qwen2.5-VL-7B）
  
节点3: 字段提取
  ├─ 规则层（身份证、户口本、结婚证等）
  ├─ VLM层（离婚证、未知类型）
  └─ LLM层（购房合同、存量房合同、房产证）❌ 失败
```

### 5.2 优化后架构

```
节点1: OCR文本提取
  └─ PP-OCRv6（唯一）✅ 简化
  
节点2: 文档分类
  └─ 规则分类器（唯一）✅ 简化
  
节点3: 字段提取
  ├─ 明确分类 → 按路由执行
  │   ├─ 规则层（身份证、户口本、结婚证、发票等）
  │   └─ VLM层（离婚证）
  └─ 未知类型 → VLM层（通用Prompt）✅ 替代LLM层
  
备注：
- LLM层（PP-ChatOCRv4）重新定位为兜底方案
- 仅在VLM提取失败时调用
- 合同类文档改用VLM层多页提取
```

### 5.3 多页VLM提取方案

**核心思路**：
```
多页文档（N张图片）
  ↓
[步骤1] 确定目标字段列表
  - 购房合同: 合同编号、买受人、出卖人、总价款、签订日期、房屋地址、建筑面积
  - 存量房合同: 合同编号、买受人、出卖人、总价款、签订日期、房屋地址、建筑面积
  - 房产证: 证书号、权利人、共有情况、不动产单元号、房屋地址、建筑面积、用途
  ↓
[步骤2] 逐页 VLM 提取（固定字段列表）
  for page in pages[:max_pages]:  # 限制处理页数（如15页）
    fields = vlm_extract(page, key_list)
    # VLM Prompt: "请从图片中提取以下字段：合同编号、买受人、出卖人...，不存在的返回空字符串"
  ↓
[步骤3] 字段合并（取第一个非空值）
  merged_fields = {}
  for page_fields in all_pages_fields:
    for key, value in page_fields.items():
      if value and key not in merged_fields:
        merged_fields[key] = value
  ↓
[步骤4] 返回合并结果
  return merged_fields
```

**预期效果**：
- 处理速度：从 197-316 秒/文档 → 50-100 秒/文档（5页 × 10-20 秒/页）
- 提取成功率：从 0% → 70%+（参考 VLM 在其他文档上的表现）
- 字段完整性：通过多页合并，覆盖大部分关键字段

## 六、关键结论

### 6.1 LLM层问题诊断

**问题**：PP-ChatOCRv4 + Qwen35-4B 处理合同文档完全失败（0字段）

**根因**：**方案错配**
- PP-ChatOCRv4 是 RAG 问答系统，适合开放式问答
- 合同提取是固定字段提取，不需要语义检索
- 向量检索 + LLM 问答的架构过于复杂，导致速度慢、失败率高

**正确定位**：
- PP-ChatOCRv4 应该是**复杂文档的兜底方案**
- 适用于：非结构化文档、需要推理的提取、开放式问答
- 不适用于：固定字段的批量提取、多页文档的结构化提取

### 6.2 架构优化方向

**简化原则**：
1. ✅ 去掉不必要的备用方案（PaddleOCR-VL、VLM分类兜底）
2. ✅ 合同类文档改用 VLM 层多页提取
3. ✅ LLM 层重新定位为兜底方案（仅在 VLM 失败时调用）

**预期收益**：
- 处理速度：提升 2-3 倍
- 提取成功率：从 0% → 70%+
- 架构复杂度：大幅降低
- 维护成本：减少
