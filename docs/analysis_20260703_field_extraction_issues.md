## 2. 根因分析（第一性原理）

### 2.1 本质问题

**核心问题**: 正则表达式模式与实际文档格式不匹配

**分解问题**:
1. OCR识别质量问题（部分字段OCR错误）
2. 正则表达式模式过于严格（无法匹配文档变体）
3. 文档格式差异（不同案例使用不同表述）

### 2.2 逐字段分析

#### ❌ 字段1: 签署日期

**OCR文本**:
```
第一条甲、乙双方于2026年1月12签订《存量房买卖合同》
```

**当前正则**:
```python
r'(?:签署日期|签订日期)\s*[:：]?\s*(\d{4}年\d{1,2}月\d{1,2}日)'
```

**问题诊断**:
1. ❌ OCR错误: "2026年1月12签订" 缺少 "日" 字
2. ❌ 格式差异: 文档中没有 "签署日期：" 标签，而是 "于...签订" 格式
3. ❌ 正则过严: 要求必须有 "日" 字和标签

**根因**: OCR识别错误 + 正则表达式未覆盖 "于XXXX年X月X签订" 格式

---

#### ❌ 字段2: 网上签约备案合同号

**OCR文本**:
```
网上签约备案合同号为：BBJZ-2026-0112065
```

**当前正则**:
```python
r'(?:网上签约备案合同号|备案合同号|合同号)\s*[:：]?\s*([A-Z0-9\-()（）]+)'
```

**问题诊断**:
1. ✅ 关键词存在: "网上签约备案合同号"
2. ❌ 格式差异: 文档使用 "为：" 而正则只匹配 "：" 或 ":"
3. ✅ 合同号格式: "BBJZ-2026-0112065" 符合 `[A-Z0-9\-]+`

**根因**: 正则表达式未处理 "为：" 的情况

## 4. 解决方案（批判性思维）

### 4.1 方案对比

#### 方案A: 优化正则表达式（推荐）

**优点**:
- ✅ 实现简单，改动小
- ✅ 性能高，速度快
- ✅ 可立即修复4个字段

**缺点**:
- ❌ 无法解决OCR质量问题
- ❌ 需要持续维护（应对更多格式变体）
- ❌ 可能引入误匹配

**实现**:
1. 签署日期: 添加 "于XXXX年X月X签订" 模式
2. 网上签约备案合同号: 处理 "为：" 格式
3. 房屋地址: 添加 "位于xxx" 模式
4. 购房款(小写): 处理顺序反转问题
5. 不动产权证号: 测试并修复正则

**预期效果**: 完成率从46%提升到77%（10/13）

---

#### 方案B: VLM兜底（补充方案）

**优点**:
- ✅ 可解决OCR质量问题
- ✅ 适应性强，无需维护正则
- ✅ 可处理复杂场景

**缺点**:
- ❌ 速度慢（20-30秒/字段）
- ❌ 成本高（API调用费用）
- ❌ 需要VLM服务支持

**实现**:
- 对规则层无法提取的字段，调用VLM进行补充提取
- 仅对缺失字段调用，不是全量重新提取

**预期效果**: 完成率从77%提升到92%（12/13）

---

#### 方案C: 多模式融合（最优方案）

**思路**: 规则层优先 + VLM兜底 + 结果合并

**实现**:
1. 第1层: 规则层提取（快速、低成本）
2. 第2层: 字段校验（检查完整性、格式）
3. 第3层: VLM兜底（对失败字段重新提取）
4. 第4层: 结果合并（优先规则层，VLM补充）

**优点**:
- ✅ 兼顾速度和准确度
- ✅ 成本可控（仅对失败字段调用VLM）
- ✅ 适应性强

**缺点**:
- ❌ 实现复杂
- ❌ 需要VLM服务支持

**预期效果**: 完成率从46%提升到92%（12/13）
