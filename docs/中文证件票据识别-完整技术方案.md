## 一、项目概述与目标

### 1.1 需求定义

搭建一个**完全离线**的 Web API 服务，对各类中文证件和票据进行结构化字段提取。

### 1.2 目标文档类型

| 类别 | 文档类型 | 数量 |
|------|---------|:----:|
| **标准证件** | 身份证、驾驶证、行驶证、银行卡、户口本、结婚证、毕业证 | 7 |
| **标准单证** | 提单、保险单、快递单、发票、病历、处方、发货单、签收单 | 8 |
| **合同协议** | 购房合同、租房合同、房产证/不动产证、公证书、委托书、资金监管协议 | 6 |
| **合计** | | **21** |

### 1.3 约束条件

| 约束 | 说明 |
|------|------|
| **完全离线** | 不能使用任何云端 API（百度千帆等），所有模型本地部署 |
| **硬件** | 当前仅 CPU，方案需兼容未来 GPU 扩展 |
| **图像质量** | 混合输入 — 既有高清扫描件也有手机拍照件 |
| **接口** | HTTP API，支持图片上传和 JSON 结构化结果返回 |

### 1.4 分等级精度目标

> 核心原则：不同文档类型的格式复杂度差异巨大，用统一的 99.9% 要求所有类型既不现实也不经济。

| 等级 | 精度目标 | 适用文档类型 | 判定依据 |
|:----:|:-------:|------------|---------|
| **A 级** | **99.9%** | 身份证、结婚证、离婚证、驾驶证、行驶证、银行卡 | 格式全国统一，字段固定，部分有校验机制（身份证校验位） |
| **B 级** | **99%** | 发票、户口本、快递单、毕业证、发货单、签收单 | 格式较固定，但各省/机构有差异 |
| **C 级** | **95%** | 购房合同、租房合同、房产证、提单、保险单、公证书、委托书、资金监管协议 | 格式不统一，需要语义理解 |
| **D 级** | **90%** | 病历、处方 | 手写体 + 医学术语，OCR 本身精度有限 |

### 1.5 精度保障的分层策略

```
目标：字段级准确率

┌──────────────────────────────────────────────────────┐
│ 第 1 层：文档预处理（拉高 OCR 精度天花板）              │
│   → 方向矫正 + 去畸变 + 增强                          │
│   → 预期提升：拍照件 +10-15%                          │
├──────────────────────────────────────────────────────┤
│ 第 2 层：OCR 引擎（基础精度）                          │
│   → PP-StructureV3（快速）/ PaddleOCR-VL（高精度）    │
│   → 预期精度：印刷体 95%+，手写体 80%+                │
├──────────────────────────────────────────────────────┤
│ 第 3 层：结构化提取（规则/LLM/双引擎）                 │
│   → A/B 级：规则解析器（快+准+无幻觉）                 │
│   → C 级：LLM 提取器（语义理解）                      │
│   → D 级：LLM + 领域知识                              │
├──────────────────────────────────────────────────────┤
│ 第 4 层：格式校验（兜底验证）                          │
│   → 身份证校验位、日期合法性、金额一致性               │
│   → 预期拦截：50% 的 OCR 错误                        │
├──────────────────────────────────────────────────────┤
│ 第 5 层：置信度评估 + 人工复核                         │
│   → 低置信度自动标记 → 人工复核                       │
│   → 最终兜底，确保端到端准确率                        │
└──────────────────────────────────────────────────────┘
```

## 二、技术方案对比与选型

### 2.1 方案总览

本项目评估了 PaddleOCR 生态中的 4 条技术路线：

| 方案 | 技术本质 | 模型大小 | CPU 速度 | 结构化提取 |
|------|---------|:-------:|:-------:|:---------:|
| **A. PP-StructureV3** | 经典 OCR 管线（多专用小模型拼接） | ~600 MB | 快（0.5-2 秒/张） | ❌ 不做 |
| **B. PaddleOCR-VL v1.6** | VLM 一体化（版面分析 + 0.9B VLM） | ~2 GB | 慢（3-8 秒/张） | ❌ 不做 |
| **C. PP-ChatOCRv4** | OCR + LLM 信息抽取（文心大模型） | 多个模型 + 云 API | 中等 | ✅ 直接做 |
| **D. 混合架构（我们的方案）** | OCR + 规则/LLM 混合提取 | ~2.5 GB（双引擎） | 可变（0.2-8 秒） | ✅ 直接做 |

### 2.2 能力维度对比

| 对比维度 | PP-StructureV3 | PaddleOCR-VL v1.6 | PP-ChatOCRv4 | 混合架构（我们） |
|---------|:-------------:|:-----------------:|:------------:|:-------------:|
| **文本识别（印刷体中文）** | 86.38% Acc（PP-OCRv5） | VLM 端到端（~95%+） | 同左 OCR 层 | 可选引擎 |
| **文本识别（手写体）** | PP-OCRv5 有限支持 | v1.6 增强 | 同左 OCR 层 | PaddleOCR-VL |
| **表格识别** | SLANet_plus 专用模型（63.69%） | VLM 端到端（SOTA 96.3%） | SLANet_plus | 可选引擎 |
| **印章识别** | 独立检测+识别 | v1.5+ 支持 | 独立检测+识别 | 两种都支持 |
| **生僻字/古籍** | 有限 | v1.6 大幅增强 | 有限 | PaddleOCR-VL |
| **多语言** | 46 种 | 109 种 | 46 种 | PaddleOCR-VL |
| **结构化字段提取** | ❌ 需后处理 | ❌ 需后处理 | ✅ 内置（keys 参数） | ✅ 规则+LLM |
| **信息抽取方式** | 无 | 无 | LLM（文心 API） | 规则+LLM（本地） |
| **RAG 长文档处理** | ❌ | ✅ restructure_pages | ✅ build_vector | 可实现 |
| **文档预处理** | ✅ 方向+矫正 | ✅ 方向+矫正 | ✅ 方向+矫正 | ✅ 两种都支持 |
| **CPU 推理速度** | **快（0.5-2 秒）** | 慢（3-8 秒） | 中等 | 快-慢可变 |
| **模型总大小** | **小（~600 MB）** | 中（~2 GB） | 大（多模型+API） | 中（~2.5 GB） |
| **内存占用** | **低（~1-1.5 GB）** | 中（~2-3 GB） | 高 | 中（按需加载） |
| **微调能力** | **几乎所有模块** | 仅 VLM（ERNIEKit） | 所有模块 | 继承 |
| **输出格式** | JSON/MD/Word/HTML/XLSX | JSON/MD/Word | JSON | 自定义 |
| **离线运行** | ✅ | ✅ | ❌ 需云 API（除非本地部署文心） | ✅ |
| **双引擎交叉验证** | ❌ | ❌ | ❌ | ✅ 可实现 |
| **GPU 加速潜力** | ✅ | ✅（<1 秒/张） | ✅ | ✅ |

### 2.3 速度-精度-资源三维对比

| 方案 | CPU 单张耗时 | GPU 单张耗时 | 模型磁盘 | 运行时内存 | 字符级精度 |
|------|:----------:|:----------:|:-------:|:--------:|:---------:|
| PP-StructureV3 精简模式 | ~0.2 秒 | ~0.05 秒 | ~30 MB | ~500 MB | ~90% |
| PP-StructureV3 标准模式 | ~1.5 秒 | ~0.3 秒 | ~600 MB | ~1.5 GB | ~95% |
| PP-StructureV3 完整模式 | ~3 秒 | ~0.5 秒 | ~600 MB | ~2 GB | ~96% |
| PaddleOCR-VL v1.6 | ~5-8 秒 | ~0.5-1 秒 | ~2 GB | ~3 GB | ~96%+ |
| PP-ChatOCRv4（仅 LLM） | ~10-20 秒 | ~3-5 秒 | 多模型 | ~4 GB+ | ~95%+ |

### 2.4 选型结论

**不是"二选一"，而是"双引擎按需调度"。**

| 场景 | 推荐引擎 | 原因 |
|------|---------|------|
| 固定格式证件（A 级） | PP-StructureV3 | 速度快 5 倍，精度足够，规则校验兜底 |
| 表格密集型（B 级） | PP-StructureV3 | SLANet 表格专用模型，输出 HTML/XLSX |
| 复杂/手写/多语言（C/D 级） | PaddleOCR-VL | 端到端识别更强，手写体增强 |
| 交叉验证需求 | 双引擎 | 两引擎结果比对，不一致时触发人工 |

## 三、各方案概要总结

### 3.1 PP-StructureV3 — 经典 OCR 管线

**一句话总结**：由多个专用小模型（文本检测/识别、版面分析、表格识别、印章识别等）组成的模块化 OCR 管线，每个模块可独立替换和微调。

**核心优势**：
- ⚡ 速度快：CPU 0.5-2 秒/张，GPU <0.5 秒
- 💾 资源省：全部模块 <600 MB
- 🔧 可微调：几乎所有模块都能用自有数据微调
- 📊 表格强：SLANet_plus 专用表格模型，输出 HTML/XLSX
- 🏗️ 模块化：哪个模块不好就替换/微调哪个

**核心局限**：
- ❌ 只做 OCR，不做信息抽取
- ❌ 手写体识别有限
- ❌ 多语言支持不如 VLM（46 vs 109 种）
- ❌ 生僻字/古籍识别弱于 VLM

**适用场景**：格式固定的证件（身份证/结婚证/发票等）、表格密集型文档、对速度要求高的批量处理。

### 3.2 PaddleOCR-VL v1.6 — VLM 一体化管线

**一句话总结**：版面分析 + 0.9B 视觉语言模型的端到端文档解析方案，OmniDocBench 精度 96.3%，支持 109 种语言。

**核心优势**：
- 🏆 精度高：OmniDocBench 96.3% SOTA
- 🌍 多语言：109 种语言
- ✍️ 手写强：v1.6 大幅增强手写体识别
- 📄 多页合并：restructure_pages 支持跨页表格合并
- 🔤 生僻字：v1.6 古籍/生僻字增强

**核心局限**：
- 🐢 速度慢：CPU 5-8 秒/张
- 💾 资源重：模型 ~2 GB
- 🔧 微调受限：仅 VLM 可微调，版面分析不可
- ❌ 同样不做信息抽取
- ❌ 无 HTML/XLSX 表格输出

**适用场景**：手写体文档、多语言文档、生僻字文档、长篇多页合同、复杂排版文档。

### 3.3 PP-ChatOCRv4 — OCR + LLM 信息抽取

**一句话总结**：飞桨官方的文档智能分析方案，结合 OCR + LLM（文心大模型）直接实现指定字段提取，是最接近"开箱即用"的信息抽取方案。

**核心优势**：
- ✅ 开箱即用：传 keys 直接返回 JSON
- 🧠 RAG 长文档：build_vector 支持超长文档
- 🔄 双通道融合：OCR→LLM + 图片→MLLM 融合
- 📝 Prompt 工程完善：任务描述+规则+格式+Few-shot

**核心局限**：
- ❌ 默认需要百度云 API（不满足离线要求）
- ❌ 本地部署文心大模型需要 GPU
- ❌ 全部走 LLM，固定证件效率低
- ❌ 没有规则解析层

**与我们方案的关系**：
思路一致（OCR + LLM 提取），但我们用本地 Qwen2.5 替代云端文心，并增加了规则解析层。PP-ChatOCRv4 的 **Prompt 工程（任务描述+规则+格式+Few-shot）、RAG 长文档处理、文本/表格分别处理**等设计理念值得借鉴。

### 3.4 混合架构（我们的方案）

**一句话总结**：双 OCR 引擎（PP-StructureV3 + PaddleOCR-VL）按需调度 + 规则/LLM 混合提取 + 格式校验 + 置信度评估 + 人工复核兜底。

**核心优势**：
- 🎯 分而治之：不同文档类型用最适合的引擎
- ⚡ 速度可调：固定证件走快速通道，复杂文档走高精度通道
- ✅ 完全离线：所有模型本地部署
- 🔄 交叉验证：双引擎结果比对提升置信度
- 📈 渐进增强：CPU 先用，未来加 GPU 自动加速

**核心局限**：
- 🏗️ 复杂度高：需要维护两套引擎+多种提取器
- 📊 需要更多测试数据来验证各引擎的实际表现
- 🧩 集成工作量大

## 四、应用场景与方案建议

### 4.1 标准证件识别

| 证件类型 | 精度目标 | 推荐方案 | 提取方式 | 关键校验 |
|----------|:-------:|---------|---------|---------|
| **身份证** | A 级 99.9% | PP-StructureV3 | 规则解析器 | 18 位校验位验证 |
| **驾驶证** | A 级 99.9% | PP-StructureV3 | 规则解析器 | 证号格式校验 |
| **行驶证** | A 级 99.9% | PP-StructureV3 | 规则解析器 | 车牌号+VIN 校验 |
| **银行卡** | A 级 99.9% | PP-StructureV3 | 规则解析器 | Luhn 算法校验卡号 |
| **结婚证** | A 级 99.9% | PP-StructureV3 | 规则解析器 | 字号格式校验 |
| **户口本** | B 级 99% | PP-StructureV3 + 表格识别 | 规则解析器 | 身份证号校验 |
| **毕业证** | B 级 99% | PP-StructureV3 | 规则解析器 | 证书编号校验 |

**方案总结**：标准证件全部走 PP-StructureV3 快速通道。利用规则解析器 + 格式校验（校验位/Luhn 等）达到 A 级精度。PP-StructureV3 的速度优势（0.5-2 秒/张）在此场景充分发挥。

### 4.2 标准单证识别

| 单证类型 | 精度目标 | 推荐方案 | 提取方式 | 难点 |
|----------|:-------:|---------|---------|------|
| **发票** | B 级 99% | PP-StructureV3 + 表格识别 | 规则解析器 | 金额校验（价税合计=金额+税额） |
| **快递单** | B 级 99% | PP-StructureV3 | 规则解析器 | 运单号格式因公司而异 |
| **发货单** | B 级 99% | PP-StructureV3 + 表格识别 | 规则解析器 | 表格中商品明细提取 |
| **签收单** | B 级 99% | PP-StructureV3 | 规则解析器 | 签名区域 OCR |
| **提单** | C 级 95% | PaddleOCR-VL | LLM 提取器 | 中英混合，多语言场景 |
| **保险单** | C 级 95% | PaddleOCR-VL | LLM 提取器 | 格式不统一，条款复杂 |
| **病历** | D 级 90% | PaddleOCR-VL | LLM + 人工 | 手写体+医学术语 |
| **处方** | D 级 90% | PaddleOCR-VL | LLM + 人工 | 手写体，OCR 精度本身有限 |

**方案总结**：格式固定的单证（发票/快递单/发货单/签收单）走 PP-StructureV3 + 规则；格式复杂或涉及多语言的（提单/保险单）走 PaddleOCR-VL + LLM；手写体为主的（病历/处方）走 PaddleOCR-VL + LLM + 人工复核。

### 4.3 表格识别

| 表格类型 | 精度目标 | 推荐方案 | 原因 |
|----------|:-------:|---------|------|
| **有线表格（发票/报表）** | B 级 99% | PP-StructureV3 | SLANet 有线表格专用模型，输出 HTML |
| **无线表格（清单/明细）** | B 级 99% | PP-StructureV3 | SLANet_plus 无线表格增强 |
| **合并单元格表格** | B 级 99% | PP-StructureV3 | SLANet_plus 支持合并单元格 |
| **跨页表格（多页合同附表）** | C 级 95% | PaddleOCR-VL | restructure_pages 跨页合并 |
| **拍照歪斜表格** | B 级 99% | 两种均可（先矫正再识别） | 文档预处理必不可少 |

**方案总结**：表格识别是 PP-StructureV3 的强项（SLANet_plus 专用模型），且支持 HTML/XLSX 输出，对后续结构化提取更友好。跨页表格场景用 PaddleOCR-VL 的 restructure_pages。

### 4.4 长文本识别

| 文档类型 | 页数 | 精度目标 | 推荐方案 | 原因 |
|----------|:----:|:-------:|---------|------|
| **购房合同** | 10-50 页 | C 级 95% | PaddleOCR-VL + restructure_pages | 跨页表格合并+多页结果合并 |
| **租房合同** | 3-10 页 | C 级 95% | 两种均可 | 文本量适中 |
| **资金监管协议** | 5-15 页 | C 级 95% | PP-StructureV3 | 表格为主，SLANet 强 |
| **公证书** | 3-10 页 | C 级 95% | PP-StructureV3 | 格式较固定 |
| **长篇 PDF（任意）** | 10+ 页 | C 级 95% | PaddleOCR-VL | 多页处理能力强 |

**方案总结**：长篇多页文档优先 PaddleOCR-VL（restructure_pages 支持跨页合并）。中等长度的文档 PP-StructureV3 速度更快。如果未来有 GPU，PaddleOCR-VL 的多页处理优势更明显。

### 4.5 手写识别

| 文档类型 | 手写比例 | 精度目标 | 推荐方案 | 现实评估 |
|----------|:-------:|:-------:|---------|---------|
| **处方** | 80%+ | D 级 90% | PaddleOCR-VL + 人工 | ⚠️ 纯 OCR 可能只能到 70-80%，必须人工兜底 |
| **病历** | 50-80% | D 级 90% | PaddleOCR-VL + 人工 | ⚠️ 手写体+术语，建议高比例人工 |
| **签名** | 100% | — | 不做 OCR | 签名识别是独立领域，不在本方案范围 |
| **手写表格** | 50%+ | D 级 90% | PaddleOCR-VL + 人工 | ⚠️ 建议人工为主，OCR 辅助 |

**方案总结**：手写识别是所有 OCR 技术的难点。PaddleOCR-VL v1.6 虽然增强了手写体，但在处方/病历场景下，**纯自动化难以达到 90%**。建议：OCR 做初步识别 + 高比例人工复核。这是 D 级精度目标的现实。

### 4.6 综合方案矩阵

| 场景 | 首选引擎 | 提取方式 | 校验机制 | 人工兜底 |
|------|---------|---------|---------|:-------:|

### 5.2 双引擎按需加载策略

```python
# 核心逻辑伪代码
def process_image(image):
    # Step 0: 预处理
    image = preprocess(image)  # 方向矫正 + 去畸变
    
    # Step 1: 快速分类（用 PP-StructureV3 mobile 模式，~200ms）
    quick_ocr = structure_v3.predict(image, mode="mobile")
    doc_type = classify_document(quick_ocr.text)
    
    # Step 2: 根据文档类型选择引擎
    if doc_type in A_LEVEL_TYPES or doc_type in B_LEVEL_TYPES:
        # A/B 级：PP-StructureV3 标准模式
        ocr_result = structure_v3.predict(image, mode="standard")
    else:
        # C/D 级：PaddleOCR-VL 高精度模式
        ocr_result = paddleocr_vl.predict(image)
    
    # Step 3: 结构化提取
    if doc_type in RULE_BASED_TYPES:
        fields = rule_extractor.extract(ocr_result, doc_type)
    else:
        fields = llm_extractor.extract(ocr_result, doc_type)
    
    # Step 4: 校验 + 置信度
    fields, confidence = validate_and_score(fields, doc_type)
    
    # Step 5: 路由
    if confidence < THRESHOLD:
        return {"fields": fields, "confidence": confidence, "need_review": True}
    else:
        return {"fields": fields, "confidence": confidence, "need_review": False}
```

## 六、精度保障体系

### 6.1 精度分层与保障策略

| 等级 | 目标 | 保障策略 | 预期达成路径 |
|:----:|:----:|---------|------------|
| A 级 99.9% | 身份证/驾驶证等 | 规则解析 + 校验位验证 + OCR 置信度筛选 | 规则解析器 100%（前提是 OCR 正确）+ 校验位拦截 50% OCR 错误 → 剩余错误率极低 |
| B 级 99% | 发票/户口本等 | 规则解析 + 交叉字段校验 + 表格结构化 | 表格 HTML 输出提升结构化精度 + 金额等数值交叉验证 |
| C 级 95% | 合同/协议等 | LLM 提取 + Prompt 工程 + 格式校验 | 结构化 Prompt（任务+规则+格式+Few-shot）提升 LLM 提取准确率 |
| D 级 90% | 病历/处方 | LLM + 领域词典 + 高比例人工 | OCR 做初步识别 + 人工复核兜底 |

### 6.2 格式校验规则清单

| 文档类型 | 校验字段 | 校验规则 | 拦截能力 |
|----------|---------|---------|:-------:|
| 身份证 | 身份证号 | 18 位校验位（ISO 7064:1983） | 强 |
| 银行卡 | 卡号 | Luhn 算法 | 强 |
| 发票 | 金额 | 价税合计 = 金额 + 税额 | 强 |
| 发票 | 发票代码/号码 | 位数 + 数字格式 | 中 |
| 所有 | 日期 | 年月日合法性 + 范围检查 | 中 |
| 所有 | 电话 | 位数 + 区号格式 | 中 |
| 结婚证 | 结婚证字号 | 格式正则（J+地区+年份+序号） | 中 |

### 6.3 置信度评估模型

```
综合置信度 = w1 × OCR 置信度 + w2 × 规则匹配度 + w3 × 格式校验通过率 + w4 × 引擎一致性

其中：
- OCR 置信度：引擎返回的平均 rec_score
- 规则匹配度：正则表达式匹配成功的字段比例
- 格式校验通过率：通过格式校验的字段比例
- 引擎一致性：（双引擎模式下）两个引擎提取结果的一致性

阈值设定：
- A 级文档：置信度 < 0.95 → 人工复核
- B 级文档：置信度 < 0.90 → 人工复核
- C 级文档：置信度 < 0.80 → 人工复核
- D 级文档：全部人工复核（OCR 辅助）
```
