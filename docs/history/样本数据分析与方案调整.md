## 三、问题根因分析

### 3.1 为什么字段匹配率这么低？

#### 问题1：OCR的模板提取不完整

**典型案例**：结婚证

```
OCR提取结果：
- 识别出"民政部"等关键词
- 但字段提取为空：{}

VLM提取结果：
- 持证人：XXX
- 姓名_男方：XXX
- 姓名_女方：XXX
- 登记日期：XXX
- 结婚证字号：XXX
```

**原因**：OCR的模板规则不完善，只识别了文档类型，没有提取具体字段。

#### 问题2：户口本字段复杂

**典型案例**：户口本

```
OCR提取结果：
- 与户主关系：户主关系性民曾用名（错误）
- 户主：关系性民曾用名（错误）

VLM提取结果：
- 与户主关系：子（正确）
- 户主：张艳芳（正确）
- 姓名：李关来（正确）
```

**原因**：户口本包含多成员信息，表格结构复杂，OCR的模板无法处理。

#### 问题3：购房合同/房产证字段多

**典型案例**：购房合同

```
OCR提取结果：
- 字段提取为空：{}

VLM提取结果：
- 买受人：XXX
- 总价款：XXX
- 出卖人：XXX
- 房屋坐落：XXX
- 建筑面积：XXX
```

**原因**：合同类文档字段繁多（10-20个），OCR的简单模板无法覆盖。

### 3.2 为什么响应时间超过20秒？

#### 分析

| 阶段 | 耗时 | 优化空间 |
|------|:----:|:--------:|
| 图片加载 | 0.1s | - |
| OCR文本检测 | 2-5s | ⚠️ 可优化 |
| OCR文本识别 | 3-8s | ⚠️ 可优化 |
| 版面分析 | 2-5s | ⚠️ 可优化 |
| 表格识别 | 2-5s | ⚠️ 可优化 |
| 模板匹配 | <0.1s | - |
| VLM推理 | 15-30s | ❌ 难以优化 |
| **总计** | **23-50s** | - |

**瓶颈**：
1. OCR管线包含多个模型（检测+识别+版面+表格），串行执行
2. VLM推理本身就很慢（0.9B参数）
3. 没有针对CPU优化的量化版本

## 五、风险与应对

### 风险1：VLM速度无法满足<20秒要求

**可能性**：高（当前VLM平均28.9秒）

**应对方案**：
1. 优先使用规则层处理固定文档（<1秒）
2. 对复杂文档，使用PP-ChatOCRv4（预估15-60秒）
3. 如果仍然超时，考虑：
   - 使用更小的LLM模型（Qwen2.5-0.5B）
   - 限制处理页数（最多处理前10页）
   - 异步处理（先返回部分结果）

### 风险2：复杂文档准确率无法达到99%

**可能性**：中（当前PP-ChatOCRv4未经实测）

**应对方案**：
1. 先用PP-ChatOCRv4实测准确率
2. 如果不达标，优化Prompt工程
3. 如果仍然不达标，考虑：
   - 使用更大的LLM模型（Qwen2.5-7B）
   - 引入人工复核环节
   - 分步提取字段（先提取关键字段，再提取其他字段）

### 风险3：户口本处理效果差

**可能性**：高（当前匹配率仅20.5%）

**应对方案**：
1. 优先尝试规则层（针对户口本设计专门的规则）
2. 如果规则层效果差，引入VLM层（GLM-OCR或PaddleOCR-VL）
3. 如果VLM速度太慢，考虑：
   - 限制处理页数（只处理首页）
   - 使用更小的VLM模型
   - 异步处理

## 六、总结与建议

### 核心结论

1. **固定文档（37.5%）可以100%准确**：身份证、结婚证已有验证
2. **复杂文档（30.8%）准确率差距大**：当前仅25-35%，需要大幅优化
3. **响应时间都超标**：OCR平均23s，VLM平均29s，需要优化到<20s
4. **VLM在复杂文档上更强**：购房合同、房产证VLM明显优于OCR

### VLM和LLM选型

详见：[VLM和LLM选型分析.md](./VLM和LLM选型分析.md)

**VLM选型**：
- **推荐**：GLM-OCR（llama.cpp量化版）
- **理由**：OmniDocBench 94.62分（SOTA），视觉理解最强
- **预估速度**：30-60秒（需要优化到<20秒）

**LLM选型**：
- **推荐**：Qwen3.5-4B（首选）或 Qwen2.5-3B（备选）
- **理由**：2025年7月最新发布，性能比Qwen2.5提升15-20%
- **预估速度**：40-67秒（需要优化到<20秒）

**重要说明**：
- ❌ **Qwen3.6-12B 不存在**：Qwen3.6只有35B-A3B（MoE）
- ❌ **12-14B Dense模型在CPU上不可行**：推理速度>100秒，严重超时
- ❌ **其他厂商没有合适的4B模型**：DeepSeek最小1.5B，Kimi是MoE，GLM最小9B
- ✅ **坚持4B级别**：Qwen3.5-4B是最优选择

### 推荐方案

**方案E+（增强版三层混合架构）**：
- 规则层处理固定文档（37.5%）：100%准确，<1秒
- 规则层增强处理半固定文档（26.7%）：99%准确，<15秒
- VLM层（GLM-OCR）处理复杂文档（30.8%）：99%准确，<20秒

- **如果GLM-OCR速度30-60秒**：采用，但需要性能优化
