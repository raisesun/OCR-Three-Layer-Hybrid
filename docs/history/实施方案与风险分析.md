### 2.2 评估指标体系

```
准确率指标：
├── 字段级准确率（Field-level Accuracy）
│   ├── 完全匹配率：字段值完全一致
│   ├── 部分匹配率：允许格式差异（如日期格式）
│   └── 关键字段准确率：姓名、身份证号等加权
├── 文档级准确率（Document-level Accuracy）
│   ├── 所有字段正确：严格标准
│   └── 关键字段正确：业务可接受
└── 整体指标
    ├── 加权平均准确率（按文档类型占比）
    └── 失败率：< 1%

速度指标：
├── 单文档响应时间（P50/P90/P99）
├── 端到端处理时间
└── 并发处理能力

质量指标：
├── 人工介入率：< 5%
├── 置信度分布：高/中/低
└── 失败案例根因分析
```

#### Step 1: 建立评估框架

创建评估工具类：

```python
class OCREvaluator:
    """OCR系统评估器"""
    
    def __init__(self, ground_truth_dir):
        """
        初始化评估器
        
        Args:
            ground_truth_dir: ground truth标注目录
        """
        self.gt = self.load_ground_truth(ground_truth_dir)
        self.results = []
    
    def load_ground_truth(self, gt_dir):
        """加载ground truth标注"""
        # 从JSON文件加载
        pass
    
    def evaluate_field(self, predicted, expected, field_type):
        """
        字段级评估
        
        Args:
            predicted: 预测值
            expected: 期望值
            field_type: 字段类型（姓名、身份证号等）
        
        Returns:
            dict: {
                'match': bool,  # 是否匹配
                'score': float,  # 匹配分数（0-1）
                'error_type': str  # 错误类型
            }
        """
        # 完全匹配
        if predicted == expected:
            return {'match': True, 'score': 1.0, 'error_type': None}
        
        # 部分匹配（允许格式差异）
        if field_type == 'date':
            # 日期格式标准化
            pred_norm = self.normalize_date(predicted)
            exp_norm = self.normalize_date(expected)
            if pred_norm == exp_norm:
                return {'match': True, 'score': 0.9, 'error_type': 'format_diff'}
        
        if field_type == 'id_number':
            # 身份证号校验
            if self.validate_id_number(predicted):
                return {'match': False, 'score': 0.5, 'error_type': 'format_error'}
        
        return {'match': False, 'score': 0.0, 'error_type': 'mismatch'}
    
    def evaluate_document(self, doc_id, predictions):
        """
        文档级评估
        
        Args:
            doc_id: 文档ID
            predictions: dict, {field_name: predicted_value}
        
        Returns:
            dict: {
                'doc_id': str,
                'field_accuracy': float,  # 字段准确率
                'exact_match': bool,  # 是否完全匹配
                'key_fields_correct': bool,  # 关键字段是否正确
                'details': list  # 详细评估结果
            }
        """
        expected = self.gt[doc_id]
        details = []
        
        for field_name, pred_value in predictions.items():
            exp_value = expected.get(field_name)
            if exp_value is None:
                continue
            
            field_result = self.evaluate_field(pred_value, exp_value, field_name)
            details.append({
                'field': field_name,
                'predicted': pred_value,
                'expected': exp_value,
                **field_result
            })
        
        # 计算字段准确率
        correct_count = sum(1 for d in details if d['match'])
        field_accuracy = correct_count / len(details) if details else 0
        
        # 判断关键字段是否正确
        key_fields = ['姓名', '身份证号', '金额']
        key_correct = all(
            d['match'] for d in details if d['field'] in key_fields
        )
        
        return {
            'doc_id': doc_id,
            'field_accuracy': field_accuracy,
            'exact_match': field_accuracy == 1.0,
            'key_fields_correct': key_correct,
            'details': details
        }
    
    def generate_report(self):
        """
        生成评估报告
        
        Returns:
            dict: {
                'overall_accuracy': float,
                'exact_match_rate': float,
                'key_field_accuracy': float,
                'by_document_type': dict,
                'failure_cases': list
            }
        """
        # 统计整体指标
        total_docs = len(self.results)
        exact_match_count = sum(1 for r in self.results if r['exact_match'])
        key_field_correct_count = sum(1 for r in self.results if r['key_fields_correct'])
        
        overall_accuracy = sum(r['field_accuracy'] for r in self.results) / total_docs
        exact_match_rate = exact_match_count / total_docs
        key_field_accuracy = key_field_correct_count / total_docs
        
        # 按文档类型统计
        by_type = {}
        for r in self.results:
            doc_type = self.get_doc_type(r['doc_id'])
            if doc_type not in by_type:
                by_type[doc_type] = []
            by_type[doc_type].append(r)
        
        by_type_stats = {
            doc_type: {
                'count': len(results),
                'avg_accuracy': sum(r['field_accuracy'] for r in results) / len(results),
                'exact_match_rate': sum(1 for r in results if r['exact_match']) / len(results)
            }
            for doc_type, results in by_type.items()
        }
        
        # 失败案例
        failure_cases = [r for r in self.results if not r['exact_match']]
        
        return {
            'overall_accuracy': overall_accuracy,
            'exact_match_rate': exact_match_rate,
            'key_field_accuracy': key_field_accuracy,
            'by_document_type': by_type_stats,
            'failure_cases': failure_cases
        }
```

#### R1: 速度性能不达标 🔴

**风险描述：**
- 当前OCR管线23.3秒，VLM 30.4秒，都超过20秒要求
- 量化后可能仍无法满足实时性要求
- 加权平均耗时目标<15秒

**影响范围：** 所有文档类型

**概率评估：** 中（50%）
- 量化版GLM-OCR可能达到15秒
- 但LLM层（购房合同）可能>20秒

**影响程度：** 高
- 用户体验差
- 可能无法满足业务需求

**应对策略：**

1. **分级响应**
   - 简单文档（规则层）：<1秒
   - 中等文档（VLM层）：<15秒
   - 复杂文档（LLM层）：允许<30秒，前端展示进度

2. **异步处理**
   - 前端展示进度条
   - 后端并行处理
   - 支持中断和恢复

3. **缓存机制**
   - 相同文档缓存结果
   - 使用文档hash作为key
   - 缓存有效期24小时

4. **硬件升级**
   - 考虑GPU加速（如M系列Mac的Metal）
   - 考虑云端部署（如阿里云、腾讯云）

**验证方法：**
- 量化版实测，记录P50/P90/P99延迟
- 如果P90>20秒，触发应急方案

**应急方案：**
- 如果速度无法优化到<15秒，调整目标为<20秒
- 如果仍无法达标，考虑云端部署

---

#### R2: 准确率差距过大 🔴

**风险描述：**
- 当前字段匹配率38%，距离99%目标差距巨大
- 51.3%的案例"OCR和VLM都失败"
- 户口本匹配率仅20.5%，结婚证25%

**影响范围：** 半固定和复杂文档

**概率评估：** 高（70%）
- 当前性能与目标差距巨大
- 51.3%双失败率说明存在系统性问题

**影响程度：** 高
- 无法满足业务需求
- 可能导致重大业务错误

**应对策略：**

1. **深度分析失败案例**
   - 逐张分析51.3%双失败案例的根因
   - 分类统计失败原因（图片质量/格式差异/模型能力）
   - 制定针对性优化方案

2. **Prompt工程优化**
   - 针对每类文档设计专门prompt
   - 使用few-shot learning
   - 迭代优化prompt

3. **多模型投票**
   - 关键文档用多个模型交叉验证
   - 两个模型结果一致才通过
   - 不一致时转人工

4. **人工兜底**
   - 低置信度结果转人工审核
   - 建立人工审核界面
   - 人工修正结果用于模型优化

**验证方法：**
- 建立失败案例库，分类统计失败原因
- 每轮优化后重新测试，观察准确率提升

**应急方案：**
- 如果准确率无法提升到90%，调整目标为85%
- 如果仍无法达标，增加人工审核比例

---

#### R3: 户口本处理困难 🟡

**风险描述：**
- 当前匹配率20.5%，78.1%案例双失败
- 原因：多成员信息、表格结构复杂、各省格式差异

**影响范围：** 26.7%的业务量

**概率评估：** 高（60%）
- 户口本是最复杂的半固定文档
- 各省格式差异大

**影响程度：** 中
- 影响26.7%的业务量
- 但可以通过人工兜底

**应对策略：**

1. **表格结构识别**
   - 先用传统OCR识别表格结构
   - 定位"姓名"、"与户主关系"、"身份证号"等列
   - 识别成员数量

2. **分成员提取**
   - 逐行提取每个家庭成员信息
   - 降低单次任务复杂度
   - 提高准确率

3. **模板库建设**
   - 收集各省户口本模板
   - 匹配对应解析规则
   - 逐步完善模板库

4. **VLM+规则混合**
   - VLM提取粗粒度信息
   - 规则精修（如身份证号校验）
   - 结合两者优势

**验证方法：**
- 收集至少10个不同省份的户口本样本测试
- 观察准确率提升情况

**应急方案：**
- 如果准确率无法提升到85%，考虑人工审核所有户口本
- 如果仍无法达标，考虑与业务方协商简化需求

---

#### R4: 长文档处理限制 🟡

**风险描述：**
- 购房合同可能超过10页，文本量>10K token
- Qwen3.5-4B 的上下文窗口可能不足
- 长文档处理速度可能>100秒

**影响范围：** 15.8%的购房合同

**概率评估：** 中（40%）
- 购房合同通常10-30页
- 分页提取可能解决问题

**影响程度：** 中
- 影响15.8%的业务量
- 但可以通过分页处理

**应对策略：**

1. **分页处理**
   - 按页拆分，分别提取后合并
   - 每页单独输入LLM
   - 合并结果时去重和验证

2. **关键页识别**
   - 只处理包含关键字段的页面
   - 跳过重复信息页
   - 减少处理时间

3. **流式处理**
   - 边识别边返回，减少等待时间
   - 前端展示进度
   - 用户体验更好

4. **模型升级**
   - 如4B不够，考虑7B（需评估速度）
   - 考虑使用更大上下文窗口的模型

**验证方法：**
- 测试最长的5份购房合同
- 观察响应时间和准确率

**应急方案：**
- 如果响应时间>30秒，考虑只处理关键页
- 如果准确率<80%，考虑人工审核所有购房合同

---

#### R5: 文档分类器误判 🟡

**风险描述：**
- 关键词匹配可能误判文档类型
- 错误路由导致使用错误的处理层
- 新增文档类型需要更新规则

**影响范围：** 整体准确率

**概率评估：** 中（30%）
- 关键词匹配在大多数情况下有效
- 但边界案例可能误判

**影响程度：** 中
- 误判会导致准确率下降
- 但可以通过人工修正

**应对策略：**

1. **多特征分类**
   - 除了关键词，加入图像特征（尺寸、颜色）
   - 提高分类准确率
   - 减少误判

2. **置信度阈值**
   - 低置信度时人工确认
   - 避免误判
   - 提高鲁棒性

3. **可配置规则**
   - 分类规则外置配置
   - 易于更新
   - 支持快速迭代

4. **反馈学习**
   - 记录误判案例
   - 持续优化规则
   - 逐步提高准确率

**验证方法：**
- 测试边界案例（如身份证+户口本同框）
- 观察分类准确率

**应急方案：**
- 如果分类准确率<95%，考虑使用VLM预分类
- 如果仍无法达标，考虑人工确认所有分类结果

---

#### R6: 样本数据代表性不足 🟢

**风险描述：**
- 969张样本主要来自房产交易
- 可能缺少边缘案例（如老旧证件、损坏证件）
- 各省证件格式差异可能覆盖不全

**影响范围：** 泛化能力

**概率评估：** 低（20%）
- 样本来自真实业务，代表性较好
- 但可能缺少极端案例

**影响程度：** 中
- 可能影响泛化能力
- 但可以通过数据增强弥补

**应对策略：**

1. **数据增强**
   - 模拟不同光照、角度、质量
   - 增加样本多样性
   - 提高鲁棒性

2. **补充样本**
   - 收集更多省份、更多年份的证件
   - 覆盖更多格式差异
   - 提高泛化能力

3. **压力测试**
   - 故意加入低质量图片测试鲁棒性
   - 观察系统表现
   - 发现潜在问题

**验证方法：**
- 统计样本的地域分布、年份分布
- 观察是否覆盖充分

**应急方案：**
- 如果发现覆盖不足，补充特定省份样本
- 如果泛化能力差，增加数据增强

---

#### R7: 技术栈复杂度高 🟢

**风险描述：**
- 需要集成 PaddleOCR、GLM-OCR、Qwen3.5、PP-ChatOCRv4
- 各工具版本兼容性
- 部署环境依赖复杂

**影响范围：** 开发效率和运维成本

**概率评估：** 低（10%）
- 各工具都有成熟的部署方案
- 但集成时可能遇到问题

**影响程度：** 低
- 可能影响开发效率
- 但可以通过文档和容器化解决

**应对策略：**

1. **容器化部署**
   - Docker 封装各组件
   - 简化部署
   - 提高可移植性

2. **API解耦**
   - 各层独立服务
   - 通过API通信
   - 降低耦合度

3. **详细文档**
   - 记录每个组件的版本和配置
   - 便于维护和排查问题
   - 降低学习成本

4. **渐进式集成**
   - 先验证单组件
   - 再集成
   - 降低风险

**验证方法：**
- 编写自动化部署脚本
- 一键启动

**应急方案：**
- 如果集成困难，考虑使用云服务
- 如果维护成本高，考虑简化技术栈
